# はじめに

C++ Contractsが公に姿を現したのはC++20が最初でしたが、実際の議論が始まったのはもっと前で、C++26でContracts機能が採択されるまでには20年以上の時間がかかっています。

C++20においてContractsが最終的に削除されてしまったのは機能が未成熟だったこともありますが、人々がもつContractsのイメージがそれぞれ異なっていたことにも原因の一端があります。Contracts、契約プログラミングは機能面だけを見れば単に高級なアサートを導入するものであり、それを用いてできることというのは現状のアサートの活用事例がベースになります。単なるアサートに要求される機能には多様なものがあり、単一のアサートがそのすべてを担うことは到底叶いません。

C++26 ContractsはC++20での失敗の経験をもとにより洗練されたものになっていますが、それでもC++26プロセスの終了ギリギリまで議論が紛糾し、そのまま残るか不透明な状態にありました。今回は機能の未成熟というよりは、各自が求めるContracts機能とC++26 Contracts機能のずれ、すれ違いが明確に異議の中心でした。

C++26 Contractsの根底にある契約プログラミングの考え方と、それをC++に導入する方法、そしてその機能が達成したいことなどについては関連する提案でかなり丁寧に説明されています。しかし、だれもがそれをしっかりと読むわけではなく、単なる機能解説からはそのような設計思想を理解することは困難です。一方で、機能の根底にある考え方を知らなければ、間違った使い方をしようとして不便を感じてしまったり、その恩恵を十分に享受できない可能性があります。

本書では、読者がC++26 Contracts機能を理解し使いこなすことで、その主目的である「プログラムの正しさを向上させる」ことの恩恵を十二分に享受できるようになるために、単なる機能解説だけではなくC++26 Contractsの根底にある考え方から設計方針までを含めて解説しています。本書を読むことで、C++26 Contractsが何を目指していて何を目指していないのか、高級なアサートとどう異なるのか、「私の欲しいアサートはなぜないの？」か、そしてContractsとは何なのか、といったことを理解することができるはずです。

そのうえで機能やその設計思想に不服を持つことになるかもしれませんが、少なくとも背景を理解しないままに持つ不満よりは遥かに正当であり、正しい批判になると思います。そして、そのような理解がより良い・正確なコードを書くための一助になる事を願っています。

## 契約とは

C++の関数には型があり、関数型と呼ばれています。関数型は戻り値と引数の型くらいの情報しかないため、その関数がどういうものであるかはほとんど何もわかりません。

```cpp
double(double, double);
void(int, int, int, int);
```

関数の定義をする際には関数名が付与されます。関数名はその関数が何をするものなのかの情報を少しだけ表現できます。

```cpp
double atan2(double, double);
void draw_rect(int, int, int, int);
```

この関数が何をするものなのかは少しわかったものの、引数については何もわかりません。それぞれの引数には何をどの順番で渡すべきでしょうか？これも適切な名前を付けることによってある程度解決できます。

```cpp
double atan2(double y, double x);
void draw_rect(int x, int y, int width, int height);
```

ではこれでこの関数を正しく扱えるようになったかというとまだ足りない情報があります。それは引数の値には何を渡して良くて、何を渡してはいけないのかが分からない事です。`atan2()`の引数には任意の浮動小数点数値を渡してよいでしょうか？NaNやINFはどうでしょうか？`draw_rect()`はいずれかの整数値が負の値を渡すとどうなるでしょうか？

このような引数の制限の一部は型で表現することができます。

```cpp
finite<double> atan2(finite<double> y, finite<double> x);
void draw_rect(std::uint32_t x, std::uint32_t y, std::uint32_t width, std::uint32_t height);
```

そして、同様のことが戻り値にも言えます。戻り値には名前を付けられないので、その意味や制限を伝達するには型情報に頼る他ありません。

静的型付け言語では、引数や戻り値の意味について型で表現しておくことでコンパイル時に無効な入力を検査できるメリットがあります。型システムの表現力がより強いプログラミング言語では、より多くの事を静的に検査することができます。

しかし型だけでは完全に表現しきれない、あるいは表現したとしてもその検査を静的に行えない場合があります。上記例の`finite<double>`という型は浮動小数点数値が有限値をとることを表明しますが、実行時の値に対してコンパイル時に確定させる事は常にはできないためその構築時に何らかの検査が必要になります。他にも、`std::vector`の`[]`のように実行時のインデックスと配列サイズに対して範囲外アクセスをコンパイル時に検査することはできません。

また、関数の動作そのものも名前だけでは完全に表現しきることはできません。無効な入力に対しての動作やエラー時の動作もそうですし、上記例の`draw_rect()`がどこに矩形を描画するのかといった正常時の動作そのものも名前だけで完全に表現することは困難な場合がほとんどです。

名前でも型でも表現できないような残りの情報は最終的に何かしらのドキュメント（コメントやソースコードとは独立した文書など）で説明されることになります。そして、関数の利用者が関数を正しく扱うにはこれらの情報をすべて把握する必要があります。

ある人が関数（より広くプログラムやそのコンポーネント）を作成し別の人がそれを利用する場合、それを正しく扱うために必要なその動作についての実装者と利用者の間での共有事項・共通理解の事を、契約（Contract）と呼びます。

前述のように、このような共有事項（契約）は必ずしもコードで表現されているわけではなく、コード以外の部分で伝達される場合がほとんどです。その最たる例がドキュメントやリファレンスです。

この契約は、契約プログラミングという考え方とは無関係にプログラミングにおいて初めから存在しており、その概念を認知しているかに関わらずプログラマーは常にそれを扱わなければなりません。

### 契約は境界に生じる

プログラミングにおけるこのような契約は、責務の境界面において発生します。それはコードの中では関数やクラス、あるいはモジュールやコンポーネントの様な単位の境界であったり、より高レベルでは外部のライブラリだったり、より低レベルではABIやOSのシステムコールだったりします。

このような境界面では往々にしてその内外で内部の担当者（実装者、提供者）と外部の担当者（利用者）が分離され、それぞれの担当者が直接コミュニケーションを取ることはほぼありません。また、利用者はその実装を必ずしも確認できるわけではありません。

したがって、必然的にどこまでが実装者の責務でどこからが利用者の責務なのかを明確にする必要が生じます。契約はその責務を明確化し境界線を定義するもので、実装者と利用者の間で取り交わされる義務と便益の合意事項です。

契約を把握することで、利用者はその実装詳細に立ち入ることなく対象の動作について推論可能になります。逆に言うと、実装者は利用者が動作を十分推論可能になるように契約を明文化する必要があります。

### 契約が破られる場合

契約が破られる場合とは、契約が順守・履行されない場合の事を言います。契約違反とも言います。

契約が破られている場合はそのソフトウェア・コードについてバグがあることを意味します。契約が正しいとすれば、契約違反とは実装者と利用者の間での共通理解が失われている状態であり、実装側では正しく動作するための前提が崩れており、利用者側ではその実装の推論可能性が失われています。契約違反は通常場当たり的に復帰可能なものではなく、修正すべきプログラムのバグを示しています。

ただし、契約違反は必ずしも実装者の責任ではありません。利用者が契約に従わない使い方をしている場合は利用者の責任になりますし、契約に則った使い方をしているのに契約に従わない動作を取る場合は実装者の責任となります。これがどちらの責務になるかも契約の表現するところです。

また、契約違反はその対象がエラー・例外を返す事とは異なります。エラーや例外は実装側が通常想定しうるプログラム状態に対して定義・通告されるもので、それは契約に記述されている正常な動作の一部です。契約内の失敗ではあっても、契約違反ではありません。エラーや例外はバグではなく場合によっては復帰可能な状態であり、それを表明している契約の一部です。

仮に想定されるエラーや例外を契約違反としてしまうと、エラーや例外に対する責務や扱いが曖昧になります。

例えばファイルオープンを行う関数`file_open()`がある時

```cpp
std::fstream file_open(std::filesystem::path file_path);
```

この関数が成功するかどうかはファイルシステムの状態に強く依存します。例えばディスクが無いとかファイルシステム異常の場合に失敗する可能性があります。ファイルシステムが正常であることを契約条件として扱うことにした場合、それが違反された場合は誰の責任になるのでしょうか？利用者が悪いとも実装者が悪いとも言えず、責務の所在が宙に浮いてしまいます。

見方を変えると、この場合ファイルシステムが正常であるかどうか、すなわちファイルシステムとの契約は実装者との間で交わされていると見ることもできます。実装者がファイルシステムの正常性に関する契約を利用者との契約境界にそのまま露出させてしまった事で利用者と実装者との契約に第三者の責務が混在してしまっています。このような場合実装者は、ファイルシステムに起因する失敗可能性は関数のエラーとして仕様化することでファイルシステムとの契約違反を取り扱う必要があります。

また別の例で、`x / y`を行う関数`divide()`について考えます。

```cpp
int divide(int x, int y);
```

引数`y`が0の場合にどうするかは実装者がこの関数の振る舞いをどう定義するのかによります。`y == 0`を契約違反として扱うこともエラーとして扱うことも間違いではなく、実装選択の一部です。しかし例えば、エラーとして扱うことにして次のように記述されている場合

```cpp
/**
 * @brief x / y を計算する
 * @param x 割られる数
 * @param y 割る数、y == 0の場合にstd::invalid_argument例外を送出する
 */
int divide(int x, int y);
```

利用者側から見ると`y == 0`の場合の動作は明確に定義されており、`y`に0を渡すコードを書いたとしてもそれはバグではないことは明確でしょう。それはこの関数の正常な動作、すなわち契約の一部です。これをこのままエラーではなく契約違反として扱ってもそれは変わらず、エラーを契約違反として扱うというのはある種の論理矛盾であるともいえます。

### 責務を明確化する - 事前条件と事後条件

契約違反時の責務については、実装者と利用者どちらの責任になるかは場合によるとしていました。契約をもう少し分類することで、この責務を明確にすることができます。

あるプログラム（コンポーネント、関数）が正しく動作するために、それが実行（呼び出し）される前に満たされている必要がある契約条件の事を事前条件（Precondition）と呼びます。

同様に、事前条件が満たされてプログラムが実行された場合に常に満たされる必要がある契約条件の事を事後条件（Postcondition）と呼びます。

このように分類すると、事前条件を順守するのは利用者の責任であり、事前条件が順守されている状態で事後条件を順守するのは実装者の責任となることが明らかになります。すなわち、事前条件違反が起きた場合にその原因を修正すべきなのは利用者であり、事後条件違反が起きた場合にその原因を修正すべきなのは実装者となります。

#### 正しさの定義

プログラムの正しさ（Correctness）とは契約プログラミングを理解する上で重要な要素であり、何をもって正しいとするのかを定義しておく必要があります。

本書では、プログラムの正しさあるいは正しいプログラムの動作とは、「プログラムがその仕様通りに動作すること」と定義します。この仕様は明示的に提示されている（ドキュメントなど）場合もあれば、暗黙的な（実装者の思想、慣習、特定のドメインにおける常識など）場合もあります。ここでの仕様とはこれまで述べてきた契約の概念に一致するものでもあります。

ただし、仕様そのものの正しさはプログラムの正しさとは別問題であり、仕様の正確性は別に検証が必要となります。これについては本書では立ち入りません。

事前条件と事後条件は契約の一部であり、プログラムを正しく利用するための利用者と実装者の間の共通理解でした。仕様が正しいとして事前条件と事後条件がその仕様をすべて反映しているとすると、事前条件が全て順守され、プログラムが実行され、実行後に事後条件が全て順守されていれば、そのプログラムは正しく動作したと言えます。この意味で、正しいプログラムはその仕様の下でバグが無いということになります。

## 契約プログラミング、あるいは契約による設計

先ほどまでの契約という概念から、特に関数呼び出し境界の契約だけを考えて、その契約をさらに事前条件・事後条件とそれ以外の部分（関数の動作やエラーについてなどの仕様）に分割して事前条件と事後条件に着目します。この事前条件と事後条件を適切に設計し表明することで関数の利用者と実装者の間の責務を明文化しながらプログラム構造を設計します。これが、契約プログラミングあるいは契約による設計と呼ばれる方法論です。

事前条件と事後条件の定義は先ほどと同様で

- 事前条件: ある関数を呼び出す前に、呼び出し側（利用者）が順守するべき条件
- 事後条件: ある関数が事前条件を全て順守されて呼び出されている場合に、その呼び出しの終了後に呼び出された側（実装者）が順守するべき条件

となります。

事前条件と事後条件の責務と便益の分担も同様であり次のようになります

- 呼び出し側（利用者）は事前条件を順守する義務を負い、その便益として関数呼び出し後の事後条件が順守される保証を得る
- 呼び出された側（実装者）は事後条件を順守する義務を負い、その便益として関数呼び出し時点での事前条件が順守されている保証を得る

そして、仕様が正しいとすると関数呼び出しの前後で事前条件と事後条件の全てに違反が無ければ、その関数は正しく実行されているとみなすことができます。

契約プログラミングと契約による設計という言葉はしばしば同じ意味として扱われますが、実態としては微妙に異なる意味を持っています。

- 契約による設計: 関数呼び出し境界の利用者と実装者間の契約（事前条件と事後条件）に着目したソフトウェア設計手法
- 契約プログラミング: 言語機能等を利用してコードレベルで契約（事前条件と事後条件）を表明・検査すること

契約による設計の考え方はそれに関する機能があるかどうかに関わらず、すべてのプログラミングにおいて適用できるものです。しかし、それを支援する機能があればより強力に契約による設計を実践することができるようになり、プログラムの正しさを向上させることができます。

ただし、契約プログラミング機能を使用すれば契約による設計が成し遂げられるわけではありません。事前条件と事後条件について何も考えていなければ、契約プログラミングをしていても契約による設計にはならないでしょう。それでも何もしていない状態よりは遥かに良い状態ですが、契約プログラミング機能を十分に活用しより正しいプログラムを記述するためには、契約による設計がまず重要です。

契約による設計を行うだけならアサーションの様な仕組みは出てきませんし、なんならコードを書く必要もありません。クラスや関数等のコンポーネントのインターフェースを設計する際に事前条件と事後条件を明文化してリストアップして、それを基準にコンポーネント境界を設計することで契約による設計をしたことになります。

しかし、契約による設計だけではなんだかあまり旨味が感じられないかもしれません。結局それらの契約条件はコードにはほとんど反映できずコメントやドキュメントの形で表明するしかないため、契約による設計は結局よい仕様を書きましょう！程度の教訓でしかなく、その仕様と実装コードが同期しているかどうかやその契約が実際に順守されているかどうかを検査することはできません。

そこで言語機能としての契約プログラミング機能が必要になります。コードに事前条件と事後条件（および不変条件）をアサーションとして記述できるようにすることで、契約という仕様の一部がコードの一部として表現できるようになり、そのアサーションに指定された契約条件が順守されているかを実行時にチェックすることができるようになります。これにより、ドキュメント（仕様書）との同期の問題と、実際に順守されているかが分からない問題が解決されます。

また、事前条件と事後条件は関数を正しく扱うための利用者と実装者の間の契約であり、仕様が正しいとすると事前条件が順守されていて事後条件が満たされていれば、その関数の動作は正しいものとなります。契約による設計はこの状態を達成するための設計手法ですが、契約プログラミング機能によって実際にコード上でこのことが表現され検査されることでその正しさの確度・保証の度合いが大きく高まります。

さらには、契約プログラミング機能によるアサーションは静的解析にも活用することができる可能性があります。専用の構文で契約種別ごとのアサーションが表現されていることによって、静的解析器はそれらのアサーションを契約に関するものとして認識することができ、関数呼び出し境界におけるバグの検出に役立てることができます。また、アサーションによってコード上で表明されている事前条件と事後条件および不変条件を形式手法による関数の正しさの検証の補助情報として使用することもできます。

契約による設計を行いそれを契約プログラミング機能で表現することによって、これらの方法論・機能のメリットは最大化します。その意味ではどちらも同じことを指していると言えますが、片方を忘れていいわけではありません。

### 正しさの分割統治

最初の説明のように、契約は関数呼び出し境界よりももっと大きな単位、例えばプログラムの単位でも考えることができ、そこでも同様の事が言えるかもしれません。しかし、プログラム呼び出し時の契約は取り扱うべき事が膨大になり、簡単に考えることができないものとなります。

契約による設計の要点は、プログラム全体で正しさを考える代わりに巨大なプログラム全体を小さな正当性単位に分解して、プログラムをより小さな部分から正しさを確保しようとすることにあります。個々の正当性単位の正しさを局所的に確保して、それらの正しいプログラムコンポーネントを用いて組み上げられたプログラム全体を正しいものとすることを目指します。関数はこの正当性単位の代表的でわかりやすい一つです。

契約プログラミング（契約による設計+契約プログラミング機能）は、このようにプログラムの正しさを分割統治して扱うことによってプログラムの正しさを向上させるための方法です。

### 関数契約

契約による設計において主に取り扱う、関数についての契約のことを関数契約（Function Contracts）と呼びます。関数契約はこれまで説明してきた契約の概念に準ずるもので、関数を正しく扱うための実装者と利用者の間の共通理解であり、責務の境界面です。関数の事前条件と事後条件は関数契約に含まれます。

以降本書では、単に契約と言った場合はこの関数契約の事を指すものとします。

### 不変条件

ここまではあまり言及していませんでしたが、契約プログラミングにおいては事前条件と事後条件に加えてもう一つ重要な要素があり、それが不変条件（Invariant）と呼ばれる条件です。

不変条件とは、プログラム実行中のある範囲あるいは期間の間で常に成り立っているべき条件の事です。

事前条件と事後条件がそれぞれ関数の呼び出し時とリターン時に成立している必要がある条件（すなわち関数呼び出し境界のある瞬間に成立している必要がある条件）なのに対して、不変条件はそのような関数の内部の特定の部分や、ループの間、オブジェクトが生存している間、あるライブラリ関数ペアの呼び出しの間など、プログラム中の特定期間の状態についての条件です。

不変条件は契約とは少し立ち位置が異なるものの契約と独立した概念ではなく補完的なもので、契約プログラミングにおいては事前条件と事後条件と同様に重要です。そのため、事前条件と事後条件と同様に取り扱う必要があり、同じくアサーションによってコード中に記述して表明とチェックを行います。

不変条件の中でも、クラス不変条件とループ不変条件の2つは比較的言及されることが多いため聞き覚えがあるのではないかと思います。

クラス不変条件はクラスのオブジェクト（インスタンス）が生存期間にある間、そのオブジェクトの状態（メンバ変数/フィールド）についてクラス外部から観測される場所で成立している条件です。クラス不変条件はその型のオブジェクトがその型の意味論の下で有効であるために満たしている必要がある条件であり、これはクラスの契約（仕様）の一部です。

クラス不変条件は次の地点で成り立っている必要があります

- コンストラクタ呼び出し完了後
- publicなクラスメンバ関数の呼び出し前後
- デストラクタの呼び出し直前

ただし、クラス不変条件はメンバ関数の実行中に成り立っている必要はありません。クラス不変条件の成立はあくまでクラスの利用者から見た時のもので、クラスの実装者が保証すべきものです。クラスのメンバ関数の実装においてはメンバ関数呼び出し完了時に不変条件が成立するように実装すればよいのですが、同時にこのことは実装者の義務でもあります。

クラス不変条件はクラスのセマンティクスを確立するために必須の概念であり、クラス不変条件という言葉を知らずともクラス設計を行う際には必ず考えているはずです。クラス設計においてのカプセル化の概念は、メンバ変数を外部から保護することでクラス不変条件の維持を確実にするとともにクラス不変条件成立を実装者の責務として明確に分離するために重要になります。

データ型としてのクラスにおいてはクラス不変条件の成立はデータの正しさを保証することでもあり、クラス不変条件が成立しているクラスオブジェクトの表現するデータは（そのクラスの仕様の下で）正しい状態にあるということができます。これは契約プログラミングを実践してプログラムの正しさを向上させる際にも同時に重要です。

https://nous.web.nitech.ac.jp/individual/inuzuka/lecture/PLT/PLT07/PLT4-4up.pdf

ループ不変条件はループの間常に成立している必要がある条件であり、ループが正しく実行されるために必須の条件です。これは形式手法においてアルゴリズムの正しさを証明する際にも重要な役割を持ちます。

より細かくは、ループ不変条件はループの開始前と各イテレーションの前後で成り立つ条件です。

例えば階乗を計算する次のような関数があるとき

```cpp
auto fact(std::size_t n) -> std::size_t {
  // 事前条件
  assert(n != 0); // 例を簡単にするため
  assert(/*n!の結果がオーバーフローしないこと*/);

  std::size_t result = n;
  std::size_t m = n - 1;

  while(m != 0) {
    result *= m;
    --m;
  }

  return result;
}
```

この`fact()`内の`while`文によるループのループ不変条件は主に次の2つです

- `result * m! == n!`
    - `x!`は数学的な階乗を表す
- `0 <= m && m < n`
    - `std::size_t`の保証により`0 <= m`は常に成り立ち、実質的な条件は`m < n`のみ

もう少し細かく考えることもできますがここで重要なのはこの2点です。実行時にこの条件がループ中いつも成り立っていればこのループは正しく実行されていると言うことができ、逆に成り立たないことがあれば何かしらのバグが存在する可能性があります。

クラス不変条件とループ不変条件の2つ以外にも不変条件はプログラムの様々な場所・タイミングで現れますが、関数呼び出し境界の様な特定の場所に必ずしも現れるわけではないことによってドキュメント等に記載されることは稀で、良くてコメントによる記述が行われるかどうかです。そのため、不変条件をアサーションとしてコード中に記述しておくことは事前条件と事後条件をアサーションとして記述することよりも価値が高いと言えます。また、契約アサーションを形式手法に応用する際にも不変条件のアサーションは重要になります。

そのため、契約プログラミングを実践する際には事前条件と事後条件に加えて不変条件も考えて扱う必要があります。不変条件はプログラムの至る所に現れうるため適切に扱えるようになるには少し慣れが必要になるかもしれませんが、不変条件を意識してそれをアサーションを用いて可視化することもより正確なプログラムを記述することに繋がります。

https://qiita.com/Kokudori/items/2e4bd32abf7abea3186f

### ホーア論理

契約プログラミング・契約による設計はバートランド・メイヤーによって提案され、プログラミング言語Eiffelで実証された概念です。メイヤーによる契約による設計のアイデアは、ホーア論理という形式的検証の手法を実際のソフトウェア開発に導入しようとするものでした。

ホーア論理では、次のようにプログラムの正しさ（正当性）を定義します。

事前条件$P$が成り立っている状態でプログラム$C$を実行して、それが停止したならば事後条件$Q$が成り立つ時、プログラム$C$は$P$と$Q$に関して部分的に正当（partially correct）であるといい、次のように表します。

$$
\{ P \} C \{ Q \}
$$

この3つ組の事をホーアトリプルと呼びます。

あるプログラムについてホーアトリプルが与えられてそれが真である（ホーアトリプルが成り立つ）ならば、そのプログラムは正しいということができます。この正しさとは前に定義したようにプログラムが仕様通りに動作することであり、$P$と$Q$はプログラム$C$の仕様を反映したものです。プログラムそのものも仕様から起こされるため、ホーアトリプルとはプログラムの仕様を数学的に表現したものと見ることもできます。

なお、部分的というのはプログラムの停止性についての保証が無いことによります。停止しないプログラムでは事後条件が満たされることが無いため何も言うことができません。すなわち、ホーア論理とホーアトリプルの定義する正しさには、通常プログラムの停止性は含まれません。

ホーアトリプルはプログラムとその仕様から書き起こすことができますが、それだけでは仕様の別表現でしかありません。ホーアトリプルが成り立つことを数学的に正当な手続きのみによって証明することで、ホーアトリプルの形で表現されたプログラムの正当性を証明します。ホーア論理はその証明のための数学的な規則をまとめた論理体系であり、その規則に従ってホーアトリプルが成り立つことを証明可能であるとき、そのプログラムはその仕様の下で正しいということができます。

そして、この証明は機械的に扱うことができるため、ホーア論理はプログラムの形式的検証の理論的基盤の一つとなっています。

契約による設計はホーアトリプルを契約と解釈してプログラム設計に持ち込んだものであり、契約プログラミングによって事前条件と事後条件および不変条件がコード上で表現されるようになることで、仕様をホーアトリプルに起こすということをある程度コード上で行いながら実行時に検証することを目指すものです。これだけでも仕様に基づくプログラムの正しさを向上させることができますが、さらにその先でホーア論理を基礎とした形式手法によってプログラムの正しさを証明する土台にもなりえます（ただし、Eiffelでは形式的検証までは実践されておらず、C++26 Contractsにもその機能はありません）。

しかし、実際にはホーア論理のみで形式的検証を行うのはあまり現実的ではなく、ホーア論理を基礎とする形式的検証の手法でもホーア論理そのものが直接的に使用されているわけではありません。ホーア論理を理論的基礎として応用・発展させた手法や周辺手法の発展を取り込むことで、ホーア論理を基礎とする現代のプログラムの形式検証手法は実践されています。

例えば、Ada/SPARK、Frama-Cなどのプログラミング言語や静的解析ツールでは、事前条件と事後条件および不変条件のアサーションを活用してホーア論理に基づく関数の正しさの証明を行うことができ、なおかつ一定規模の産業応用の実績があります（ただし、事前条件と事後条件を書いておくだけで検証可能になるという訳ではありません）。C++26の契約プログラミング機能はそのような機能を持たないものの、契約プログラミング機能を活用してアサーションを積極的に書いておくと、どこかの将来でこのような形式手法を利用できるようになる時が来るかもしれません。

## 狭い契約と広い契約

C++の標準ライブラリも契約による設計を意識した形で規格における規定がなされており、その文脈では事前条件の有無によって関数契約（事前条件）は2つの種類に分類されます。

- 広い契約（Wide contracts）
    - 事前条件を持たない
    - 未定義動作を指定しない
        - 少なくとも呼び出し元の責任で未定義動作が発生することは無い
- 狭い契約（Narrow contracts）
    - 広い契約ではない契約。事前条件を持つ
    - 事前条件に違反すると未定義動作となる

この分類はまた、標準ライブラリにおいて関数に`noexcept`を付加する基準としても参照されており、無条件`noexcept`を付加するのは広い契約を持つ関数のみです。また、条件付き`noexcept`指定が行われるのも、関数が（静的な）条件付きで広い契約を持つ場合のみです。

例えば、`std::vector`の`.push_back()`や`.empty()`などの関数は広い契約を持っており、`std::vector`オブジェクトの状態によらず（初期化されている必要はある）呼び出すことができ、未定義動作に陥ることはありません（`.push_back()`は別の理由により`noexcept`指定されていませんが）。一方で、`.front()`や`.pop_back()`は`std::vector`オブジェクトが空ではないという事前条件を持ち、`operator[]`はインデックスが`size()`未満であるという事前条件を持つためこれらの関数は狭い契約を持っています。

狭い契約を持つ関数は事前条件に違反して呼び出すと未定義動作となります。

### 契約の広さ

関数を実装する際の理想は広い契約を持つ事ですが、そのような関数はむしろ稀で実際にはほとんどの関数は何かしらの事前条件を持っており、それによって狭い契約を持ちます。

関数が事前条件を持つ場合でも、防衛的プログラミングを採用することで事前条件をエラーに転換して関数の契約を広げることができます。例えば平方根を求める関数では、入力値が負の場合にそれをエラーとして報告することを選択することもできます

```cpp
double square_root(double value) {
  if (value < 0) {
    throw std::invalid_argument("Negative value passed to square_root.");
  }

  // 平方根の計算処理
  ...
}
```

このように本来狭い契約を人為的に広げることには、通常いくつか問題があります。

まず、そもそも契約を広げることができないか困難な場合があります。例えば、64ビット整数の比較関数が狭義弱順序を満たしているかどうかのように実行時コストが高すぎる場合や、文字列のnull終端のように確認できないような場合もあります。あるいは、範囲のソート済みチェックのようにチェックの計算量が場合によっては関数本体の計算量よりも大きくなる場合もあり、たとえ計算量が変わらなくても計算コストは追加でかかります。このような場合は契約を広げることは非現実的です。

広げることができたとしても、いくつか欠点があります。

契約を広げるとプログラムの欠陥を不適切に隠蔽することになる傾向にあります。不適切な使用法に対して修正を加えたり何事もなかったかのように完了したりすることで事前条件を排除しようとするのはあまり賢明とは言えません。このようなエラーの隠蔽は、関数の誤用がその発生場所（事前条件違反場所）ではなくそれを呼び出したコードを通して伝播することで全く異なる場所で無関係なエラーとして表面化するか、あるいは静かにバグり続けるなどの悪影響の原因となります。

また、上記`square_root()`の例のように、契約を広げると関数の入り口で事前条件を手動でハンドリングするコードを挿入することになり、コードの可読性や保守性を低下させます。また、このような条件分岐はテストの対象でもあるため、この悪影響はテストコードにも波及します。

そして、事前条件をエラーにするとその動作は関数契約（仕様）の一部となることで省略することはできなくなり、その関数の本質的動作の一部となります。これを変更しようとすることは関数の動作の破壊的な変更となり、このことは関数の機能拡張時にも制限を加えます。そのため、ここまで述べてきたような悪影響を取り除くことをより困難にしてしまいます。

逆に、狭い契約を持つ関数に過度に事前条件を付加して人為的にさらに狭くすることにもやはり欠点があります。関数のエッジケースの動作をその関数の自然な実装でサポート可能である場合、その動作は関数の正式な動作の一部とした方が（事前条件として扱うよりも）通常効率的で保守性も高くなります。

### 狭い契約における未定義動作と実行時検査

標準ライブラリにおいては狭い契約を持つ関数は事前条件に違反して呼び出すと未定義動作となることが規定されています。とはいえ、標準ライブラリのものに限らず、多くの場合に関数は事前条件に違反して呼び出されるとその実行の結果として未定義動作に陥ることがほとんどです。標準ライブラリのこの規定はこの事を反映したものです。

標準ライブラリの関数が事前条件に違反して呼び出されると、ライブラリで規定されているように呼び出し時点で未定義動作となります。これを無視して実行を継続すると、関数本体内で事前条件が満たされないことによる何らかの言語定義の未定義動作が発生することになります。前者をソフトUB、後者をハードUBと呼ぶことにすると、一般的に未定義動作と呼ばれて恐れられているのはハードUBであり、コンパイラが最適化に利用して深刻な事態を引き起こすのはハードUBの方です。ソフトUBは標準ライブラリ実装が活用するもので、通常コンパイラの最適化には利用されません。

関数呼び出し時点で生じるソフトUBの段階ではまだその未定義動作はプログラムに深刻な影響を与えませんが、ハードUBに到達するとUBによる深刻な影響が発生する可能性があります。

```cpp
// 与えられた文字列の長さを返す
// 事前条件: `str`がnull終端されている
size_t strlen(const char *str) {
  // 事前条件に違反した場合、ソフトUBは呼び出し時点で生じる
  // 実装がソフトUBに対して何もしなければ、ここではまだ何も起こらない

  size_t count = 0;

  while (str[count]) { // 👈 ハードUBはここで生じる
    ++count;
  }
  
  return count;
}
```

この例では、`str`がnull終端されていない場合にコメントで示した`str[count]`において配列の範囲外アクセスによるハードUBが発生する可能性があります。しかし、実装がソフトUBに対して何もしない場合、制御フローがそこに到達するまではプログラムの状態は有効なまま維持されています。

標準ライブラリ以外の関数はソフトUBを持たないため事前条件に違反して呼び出されても未定義動作にはなりませんが、その状態で処理を実行すると良くて間違った動作をするだけ、最悪はハードUBに到達することになります。そのため、事前条件に違反して関数が呼び出されると安全ではないが呼び出し時点ではまだ問題は生じていない、という点は標準ライブラリ関数と同様です。

契約違反時の未定義動作のこのような性質により、狭い契約を持つ関数においては事前条件を関数が呼び出された時点（関数本体の他の制御フローに移る前）で検査することでハードUBを回避して利用者の誤用を検出することができます。

ただし、これはあくまで関数契約を検査するものであり、チェック結果そのもの及びその後の動作は関数の契約（仕様）の一部ではありません。そして、このような検査は正しいプログラムにおいては実行時間の増加以外の副作用が無く、プログラムのセマンティクスを変更しません。したがって、このような検査は冗長かつオプショナルなものであり、いつでも任意に無効化することができます。

契約プログラミング機能の1つである実行時検査はこのような性質をその根拠としています。

ただし、事前条件や事後条件はそのすべてが実行時に検査可能であるわけではなく、検査可能なものでも必ずしも検査することが有益である訳でもありません。例えばオブジェクト生存期間のようにチェックする方法が無いもの、比較関数が狭義弱順序を満足するかのように検査が現実的ではないもの、ソート済み範囲の検査のように関数本体の計算量を検査の計算量が上回るものなどは実行時検査があまり現実的ではありません。また、C++の型システムや型によって保証されている条件についてはわざわざ実行時にその検査を行う必要はありません。

このようなものに該当しない、実装可能かつ現実的で過度なコストがかからず、潜在的に有益な事前条件アサーションについては実行時にチェックする価値があります。

### 標準ライブラリにおける契約チェック

関数の事前条件が安全に検査可能であるという性質は標準ライブラリ実装においても活用されており、実装定義機能ではあるものの主要な標準ライブラリ実装でライブラリ関数の事前条件の実行時検査を利用することができます。

- libstdc++: `_GLIBCXX_ASSERTIONS`マクロ
- libc++: `_LIBCPP_HARDENING_MODE`マクロ
- MSVC STL: `_ITERATOR_DEBUG_LEVEL`マクロ

C++26では契約プログラミング機能によってこのような標準ライブラリの事前条件チェックが正式な機能の一部となります（詳細は後述）。C++23以前でもこれらのライブラリ実装の提供するチェック機能を使用すると同等の検査を有効化することができます。

ソフトUBは未定義動作であるため、それが起こった場合に実装は任意の動作を選択することができます。C++23までのこのような事前条件チェックは、ソフトUBを実装が活用した結果としての「チェックして停止」という動作です。

## 類似のパラダイム・概念

契約そのもの、あるいは事前条件と事後条件および不変条件はいずれも、契約プログラミングを実践しようとしたら新しく出てきた概念というわけではなく、プログラミングにおいて当初からそこに存在していたものであり、意識的にせよ無意識的にせよこれまでも取り扱っていたものです。

そのため、その採用が任意であるようなプログラミングパラダイムやスタイルなどとは異なる立ち位置にあるものですが、契約プログラミングという言葉でくくられてそれらと同列に語られていることも珍しくありません。また、それらのパラダイムの中には契約プログラミング・契約による設計と似たような事を取り扱い、また主張しているものもあります。

ここでは、契約プログラミングに類似している、あるいは近しいものを持っているプログラミングパラダイムやスタイルなどと契約プログラミングとの関連性を見てみます。

### `assert`

現在すでに利用可能なCの`assert`マクロは、現在のC++環境で契約を表明するためのデファクト的な方法です。C++26の契約プログラミング機能はこれと全く異なるものというわけではなく、契約プログラミングを実践するために`assert`をより高機能にしたものでもあります。

詳しくは2章で見ていきますが、`assert`と比較すると

- マクロではない
- 事前条件と事後条件を構文レベルで区別でき、関数宣言に記述できる
- 動作が固定化されていない
- 条件が`false`になった時の動作をカスタマイズできる

等の違いがあります。

扱うものは同じものですが、単なるマクロ置換ではない言語機能であることによって関数契約を言語上の構造として扱うようになる点が大きく異なります。

### 防衛的プログラミング

防衛的、あるいは防御的プログラミング（Defensive programming）とは、事前の仮定が崩れるような予期しない状態にプログラムが陥いることを念頭に、そのような状態に陥ったとしても正常な実行を継続するための設計手法です。可用性や安全性、セキュリティが重視されるプログラムにおいて用いられます。防衛的プログラミングの扱う予想外の事態とは悪意ある攻撃者だけでなくソフトのバグやハードの故障、仕様変更等のプログラムの実行を妨げうる様々な事態です。防衛的プログラミングの考え方をセキュリティに特化させたものがセキュアコーディングの一部になっています。

関数の入口で引数のチェックのための`if`文を連ねることが防衛的プログラミングであると良く誤解されがちですが、これは防衛的プログラミングの一側面にすぎず、防衛的プログラミングの本質は予期しない状況に陥った場合でもプログラムの正しい動作を継続させるための、より包括的な設計アプローチです。

防衛的プログラミングは契約プログラミング（契約による設計）とはかなり異なる目的を持っています。したがって、基本的に契約プログラミングとは相補的な関係にある設計手法ですが、衝突する場合もあります。

防衛的プログラミングでは、他の関数が正しくない動作を行った結果が入力されたとしても被害を受けない状態を目指すため、関数内では事前条件の順守を仮定できず、関数呼び出し側では事後条件の順守を仮定できません。したがって、関数呼び出し時の事前条件と事後条件を基本的には検査することになります。

契約プログラミングにおいては事前条件の順守は利用者の、事後条件の順守は実装者の責務として分離し、基本的にはその成立を仮定することになるため、この点の選択が防衛的プログラミングと衝突します。

また、不変条件についての取り扱いも異なり、正しいプログラムなら成り立つはずと仮定するのではなく成り立たないことがあるかもしれないとして、それが起こらないように設計して起きてしまった場合にも問題が無いように対処するのが防衛的プログラミングです。

ただし、両者の根底には契約（事前条件と事後条件）と不変条件という共通した概念があります。防衛的プログラミングは契約と不変条件の成立をどう扱うかについて契約プログラミングとは異なる立場を取った設計選択です。

契約プログラミングを前提とした場合の防衛的プログラミングにおけるこの選択は、プログラムに求められる要件やプログラムの立ち位置などによって変化します。例えば、セキュリティが重視されないプログラムにおいては契約について防衛的になる必要性は薄くなりますが、プログラムの外部（APIの境界や利用者のインターフェースなど）とやり取りする場所においてはより防衛的になる必要があります。そのため、契約プログラミングと防衛的プログラミングは二者択一のものではなく、要件あるいは契約そのものに応じた設計の選択肢となります。

C++26の契約プログラミング機能では、基本的に契約違反はバグであり復帰可能ではないという考え方に基づいているため、契約アサーションを用いて防衛的プログラミングを行うことはできません。防衛的プログラミングを選択する場合は、契約アサーションではなく`if`文によるチェックと分岐に切り替える必要があります。

防衛的プログラミングには、契約を人為的に広げた時と同じようなデメリットがあります。つまり、検査のコストが色々なところで発生し、エラーハンドリングのためのコードは可読性を低下させると同時にコードの複雑度・テスト項目を増大させ、保守性を低下させます。また、防衛的な設計や制限が開発上の制約にもなりえます。

#### 正当性検証と入力検証

防衛的プログラミングと契約プログラミングのどちらを採用すべきかは要件等によるものの、中には防衛的プログラミングを採用しなければならず単に契約としてはならない場合があります。ここではそのような検証（実行時チェック）を入力検証と呼び、契約プログラミングを採用しても良いような検証を正当性検証と呼びます。

正当性検証はその対象がプログラムの仕様と重複しており、検証そのものはプログラムの本質的な動作に影響を与えないものです。正当性検証はプログラムの信頼度が成熟したと判断した段階で削除することができます。

入力検証はプログラム外部からの入力（ローカルリソースも含めて実行時にプログラム外部から入力されうるもの）がプログラムの仕様・要件を満たしているかを検証するもので、この対象は仕様そのものではありません。入力検証に違反するような不適切な入力は本番環境においてほぼ確実に遭遇するものであるため、入力検証はプログラムの本質的な動作に不可欠なものでありプログラムがどれだけ成熟したとしても削除できないものです。

正当性検証と入力検証には大きな違いがあるものの、一見はよく似ているため非常に混同されやすい側面があります。

ある検証が正当性検証であるか入力検証であるかを判断する基準は、検証の対象がプログラムの動作中に常に真であると仮定できるか、です。仮定できない場合、それは正当性検証の仕組み（`assert`マクロや契約プログラミング機能）で検証すべきではなく、通常の制御フロー（`if`など）による入力検証とエラーハンドリングを行うべきです。

例えば、人間（ともすればインターネット経由の任意の人）が入力してくるデータについて、プログラムの想定する仕様に完璧に従ったデータを入力してくれると期待することはできません。このような検証には入力検証が必須であり、このために正当性検証の仕組みを使用してはなりません。

正当性検証にはその検証場所についてローカルに（検証対象の周辺のコードがどう使用されるかに関わらず）検証対象が常に真であると判定できるものと、文脈依存でしか判定できないものがあります。文脈依存で真となるような検証対象でも、より大きな単位（翻訳単位、ライブラリ、モジュール/パッケージ、プログラム）で文脈を考慮すると常に真であると判定できる場合もあります。文脈依存でしか判定できない検証でも、その文脈を包含する単位の中では正当性検証とすることもでき、その場合は防衛的プログラミングと契約プログラミングのどちらを採用することもできます。

一方、文脈をどう辿っても常に真となることを仮定できないような検証は入力検証であるため、それに対しては防衛的プログラミング（あるいは後述の攻撃的プログラミング）を採用するしかありません（通常そのような検証は契約的チェックすら行わない状態にしておくことはできないはずです）。防衛的プログラミングを採用する場合、その検証はアサーションのようなものではなく`if`文のような通常の制御フローを用いたエラーハンドリングを行う必要があります。

ただし、文脈依存である正当性検証は単なる1コードの開発・保守を超えて、そのようなコードのパッケージングからテスト、デプロイ、実稼働に至るまで一貫してそれが正当性検証足りえることを保証する必要があります。プログラムの規模が大きくなるほど関係者が多くなるなどしてこれは困難になり、文脈依存である検証について正当性検証とすべきかはプログラムの要件や両者の利点と欠点などを考慮したうえでのトレードオフによって決定されることになります。

#### 予防的プログラミング

防衛的プログラミングを実践すると、関数の出入り口や不変条件の成立地点において`if`チェックを細かく行う必要が出てきます。このことは防衛的プログラミングのデメリットに直結するものであり、また関数の入口で`if`チェックしまくるような誤ったイメージの原因でもあります。

防衛的プログラミングの本質はそもそもそのような悪い状態からの悪影響を最小化しようとするものであり、`if`チェックはその取り組みの後で残った最小のものとなるべきです。これは例えば、型で関数の入力を制限することや、責務を適切に分離・再配置することによって行われます。

このような防衛的プログラミングの取り組みを指して、予防的プログラミングと呼ぶことがあります。

責務の分離や型による表明などによって考えるべき事前条件と事後条件、不変条件を削減することは、ソースコード上の情報を落とすことなく実行時のアサーションを省略することにもつながるため、予防的プログラミング（すなわち防衛的プログラミング）のこのような取り組みは契約プログラミングの実践においても重要になります。

### 攻撃的プログラミング

攻撃的プログラミング（Offensive programming）とは、プログラムの異常状態が検出された場合に実行継続を考えずに速やかにプログラムを終了させるようにする考え方です。これは、バグや障害を抱えながら中途半端な状態で動いているプログラム（正しく動作せず、何が起こるか分からない）よりも、実行中に停止されたプログラム（少なくともそれ以降何も起こらない）の方が被害が少ない、という考え方に基づいています。Fail fastという言葉で表されることもあります。

攻撃的プログラミングにおいてのプログラムの異常状態とは、契約プログラミングおよび防衛的プログラミングにおいての事前・事後条件と不変条件が破られている場合の事であり、攻撃的プログラミングにおいてもこれらの概念は共通しています。攻撃的プログラミングは、契約違反が起きた場合にどうするか？という選択肢についての選択の一つであり、契約プログラミングとは必ずしも対立する考え方ではありません。

攻撃的プログラミングの実践は明示的な`if`よりも`assert`（あるいは同種のマクロ）が使用されます。C++26の契約プログラミング機能でも、契約アサーションの動作の一つとして契約違反時に終了する動作モードが用意されているため、同じ契約アサーションを用いて攻撃的プログラミングを行うことができます。

防衛的プログラミングと同様に、攻撃的プログラミングも採用するかしないかの2択ではなくプログラムに求められる要件などによって採用を検討できる選択肢です。防衛的プログラミングと攻撃的プログラミング、そして契約プログラミングが一つのプログラムの中で同居して採用されていても問題ありません。

### 表明プログラミング

表明プログラミング（Assertive programming）とは、起こるはずがないという仮定をアサーションを用いてチェックと明文化を積極的に行うプログラミングスタイルです。

表明プログラミングと攻撃的プログラミングは表裏一体であり、攻撃的プログラミングを行うとほとんどの場合に表明プログラミングをしたことにもなります。やることは同じなのですが考え方が少し異なっており、表明プログラミング独自の視点として、通常コードに隠されているプログラマの意図を読み手に伝達するコミュニケーションの手段としてアサーションを用いるという点があります。

表明プログラミングにおいても表明対象のコードの仮定とは基本的に事前・事後条件と不変条件の成立のことであり、契約プログラミングと大きく異なる概念を取り扱うわけではありません。契約や不変条件をアサーションを用いて可視化していく点は契約プログラミングでも同じです。

そのため、契約アサーションによって契約プログラミングを実践することは自動的に表明プログラミングを実践する事にもなります。C++26の契約プログラミング機能にもそれを妨げるものはありません。

また、表明違反とエラーを区別する点にも共通項があります。表明プログラミングにおいては、アサーションを用いるべきなのは通常起こりえない状態についてで、想定されうる異常状態にはアサーションによらないエラー処理（戻り値や例外など）を用いるべきとされます。これは、契約・不変条件違反とエラーは異なるものであり、前者は通常バグの存在を示しているという契約プログラミングにおける考え方と共通しています。

# C++26 Contracts

この章では、C++26の正式機能として採用されたContracts機能の解説を行います。

## 契約アサーション

C++26 Contracts機能によって、契約プログラミング（契約による設計）の主要概念である事前条件と事後条件に対応するアサーションと不変条件等を表明するためのその他のアサーションを専用の構文によってコード中に配置し、関数契約と不変条件をC++コードの一部として記述できるようになります。これらのアサーションのことを**契約アサーション（contract assertions）**と呼びます。

契約アサーションには前述の事前条件・事後条件・その他アサーションに対応して3種類のアサーションがあります。

- `pre()` : 事前条件アサーション
- `post()` : 事後条件アサーション
- `contract_assert()` : アサーション文

3種類のアサーションは基本的には同じ使用法になり、`()`の中には契約条件を表す式を指定します

```cpp
int f(const int x)
  pre (x != 1)                // 事前条件アサーション
  post(r : r == x && r != 2)  // 事後条件アサーション
{
  contract_assert(x != 3);    // アサーション文

  return x;
}
```

事前条件と事後条件アサーションは関数の宣言の直後（`requires`節が存在する場合、その後）に対して付加し、関数本体など他の場所には指定できません。`contract_assert()`は逆に`pre()/post()`の場所には配置できず、文（*statement*）の一種として関数本体内などの文が配置できる場所に配置することができます。

関数ヘッドの直後（`requires`節が存在する場合、その後）、（定義がある場合は）本体ブロックの直前、が事前条件と事後条件アサーションを配置する専用の構文スペースになります。ここでは、`pre()`と`post()`を空白区切りで任意の個数配置することができます。`pre()`と`post()`の出現順は任意であり、両方あってもどちらか片方だけしかなくてもokです。一方で、他のものはここには配置できません。

```cpp
int f1(int x) pre(x != 1) pre(x != 2) pre(x != 3) ... pre(x != 1000) ...  // ok
{
  ...
}

int f2(const int x) post(x != 1) post(x < 0) pre(x == 0) post(r: r != x)  // ok
{
  ...
}

void f3(int x) contract_assert(x != 0)  // ng、ここに配置できるのはpre()とpost()のみ
{
  ...
}

template<typename T>
void f(const T& x) requires std::integral<T> pre(x != 0) post(0 < x)  // ok
{
  ...
}
```

関数に対する事前・事後条件アサーションの指定はフリー関数に限定されたものではなく、メンバ関数でも使用可能です。

```cpp
template<typename T>
struct S {
  T t;

  int f1(int x) pre(x == 0) post(r: x != 0) // ok
  {
    ...
  }

  auto f2() const & noexcept [[attribute]] -> T requires std::integral<T> pre(t != 0) post(r: r < T(0));  // ok

  int f3(this S& self) pre(self.t == 0) post(r: r < 0); // ok
};
```

C++26時点では、仮想関数以外のタイプの関数宣言に対して事前・事後条件アサーションを指定することができます。

事後条件アサーションだけは関数の戻り値をキャプチャするための専用の構文スペースが用意されており、`post(ret: expr)`の様に条件式`expr`の前の`ret`の部分に戻り値を指すように任意の名前を指定できます。戻り値を表すこの名前は続く`expr`内で使用することができ、値としてはその関数の戻り値が取得できます。戻り値を使用しない場合は省略して、事前条件アサーションなどと同じように書くことができます。

事後条件アサーションにおけるこの結果名（戻り値名）の使用はその性質上、戻り値の無い（`void`戻り値型の）関数では使用できません。

```cpp
void f()
  post(r: ...)  // ng、戻り値の無い関数で結果名を使用できない
  post(...)     // ok、結果名を使用しない事後条件は指定可能
{
  ...
}
```

事後条件アサーションにおけるこの結果名は、事後条件アサーションが関連付けられている関数の戻り値のオブジェクト/参照を表す束縛（*result binding*）であり、言語参照（`T&`）とは異なる形の参照の様な名前になります（これは構造化束縛において導入される各束縛の名前と同じ扱いのものです）。そして、結果名の式としての値カテゴリ（`decltype(r)`）は`const`左辺値（`const T&`、`T`は戻り値型）となります。

戻り値への参照ではなく束縛としていることによって、`return`文におけるRVO等の影響を受けないようにしており、結果名はそれらが適用された後の関数戻り値を直接表しています。

なお、`pre`/`post`は`override`等と同じ文脈依存キーワードであり、`contract_assert`はキーワードとして新規追加されています。そのため、`pre`/`post`は他の名前として使用できる一方、`contract_assert`は使用できません。

```cpp
int pre(const int x)    // ok
  pre(x != 1)
  post(post: post == x) // ok
{
  int contract_assert;  // ng、これはダメ

  ...
}
```

以下、この記事では事前条件アサーションと事後条件アサーションの事を単に事前条件と事後条件と呼びます。

### 契約アサーションの述語（契約条件式）

アサーションの種類によらず、契約アサーションには何かしら契約条件を表現する条件式を指定する必要があります。この条件式の事を述語（*predicate*）と呼びます。この述語というのはアルゴリズムの文脈などで使用される言葉と同じ意味です。

述語は文脈的に`bool`変換が可能（contextually converted to `bool`）な式である必要がある他は任意のC++式（文法上の*conditional-expression*）を記述することができます。すなわち、`if`や条件演算子の条件式として使用できるものが使用できます。ただし、後述する`const`化の違いがあります。

```cpp
void f1(int* ptr) pre(ptr)  // ok、ptr != nullptr と等価
{}

void f2(std::optional<int> opt) pre(opt)  // ok、opt.has_value() と等価
{}
```

契約アサーションに述語を指定しないでおく、ということはできません、

```cpp
void f(int n)
  pre()   // ng、条件式が必要
  post()  // ng、条件式が必要
{
  contract_assert();  // ng、条件式が必要
}
```

以下、本書ではこの述語の事を**契約条件式**と呼ぶことにします。

### `const`化

3種類いずれの契約アサーションにおいても、その契約条件式内から参照される外部の変数はすべて、暗黙的に`const`化されます。

この`const`化は契約条件式の振る舞いがC++の他の部分とかなり異なる点です。これは、契約条件式の評価に伴う副作用の発生と誤った使用を防止するための措置の一つです。

契約条件式には文脈的に`bool`変換が可能であれば任意の式を記述することができます。`const`化を除けばそれ以外に特に制限は無いため、その条件式を起点として任意のC++コードを実行することができます。プログラムの契約（仕様）の順守を検査するための契約アサーションからプログラムの状態を変更することは間違った行いであるため、これをなるべく防止するためにこの`const`化の措置が取られています。

```cpp
void f() {
  std::vector<bool> vec{true, false};

  contract_assert(vec.emplace_back(true) == true);  // ng、vecはconst
}  
```

この`const`化は契約条件式の外部の変数の使用すべてに対して作用します。関数引数、ローカル変数、グローバル/名前空間スコープの変数、`static`/`thread_local`変数、NTTPや構造化束縛、そして`this`および`*this`まで、あらゆる外部の変数は`const`化されます。事後条件における結果名の値カテゴリが`const`左辺値なのも`const`化の一部です。

```cpp
int n = 0;
struct X { bool m(); };

struct Y {
  int z = 0;

  void f(int i, int* p, int& r, X x, X* px)
    pre (++n)     // ng、const左辺値を変更しようとしている（グローバル変数
    pre (++i)     // ng、const左辺値を変更しようとしている
    pre (++(*p))  // ok、ポインタの参照先には伝播しない
    pre (++r)     // ng、const左辺値を変更しようとしている
    pre (x.m())   // ng、非constメンバ関数呼び出し
    pre (px->m()) // ok、ポインタの参照先には伝播しない 
    pre ([=, &i, *this] mutable {
      ++n;        // ng、const左辺値を変更しようとしている
      ++i;        // ng、const左辺値を変更しようとしている
      ++p;        // ok、コピーキャプチャされたクロージャ型のメンバの変更（ポインタ
      ++r;        // ok、コピーキャプチャされたクロージャ型のメンバの変更（非参照
      ++this->z;  // ok、コピーキャプチャされた*thisの変更
      ++z;        // ok、コピーキャプチャされた*thisの変更
      int j = 17;

      [&]{
        int k = 34;
        ++i; // ng、const左辺値を変更しようとしている
        ++j; // ok
        ++k; // ok
      }();

      return true;
    }());

  template <int N, int& R, int* P>
  void g()
    pre(++N)      // ng、prvalueを変更しようとしている
    pre(++R)      // ng、const左辺値を変更しようとしている
    pre(++(*P));  // ok、ポインタの参照先には伝播しない

  int h()
    post(r : ++r) // ng、const左辺値を変更しようとしている
    post(r: [=] mutable {
      ++r;  // ok、コピーキャプチャされたクロージャ型のメンバの変更
      return true;
    }());

  int& k()
    post(r : ++r); // ng、const左辺値を変更しようとしている
};
```

契約条件式には任意の式を使用できるため、当然ラムダ式が使用できます。契約条件式内で宣言されたラムダ式内部で宣言された変数（コピーキャプチャも含めて）にはこの`const`化は適用されません（参照キャプチャに対しては`const`化が適用されます）。

また、この`const`化は浅いもので、深く伝播することはありません、つまり、外部のポインタを使用しているときそのポインタそのものは`const`化されていますが（`T* const`）、ポインタの参照先（`*ptr`/`ptr->~`）までは伝播しません。

仮に`const`化を解除したい場合は、`const_cast`を使用します。

```cpp
struct S {
  int m_resource;
  std::mutex m_mtx;

  // 非constのステータスチェック
  bool cond() {
    std::lock_guard _{m_mtx};

    return m_resource != 0;
  }

  int f()
    pre(this->cond())                 // ng、非constメンバ関数呼び出し
    pre(const_cast<S*>(this)->cond()) // ok
  {}
}
```

### 事後条件における関数引数の使用

事後条件から関数引数を使用することは良くあると思われますが、その場合に使用する引数は参照であるか、そうでないなら`const`宣言されていなければなりません。非参照の場合、これは暗黙`const`化とは関係なく明示的に`const`と宣言されている必要があります。

```cpp
void f(int a1, const int a2, int& a3)
  post(a1 == 0) // ng、事後条件からの非参照非const引数の使用
  post(a2 == 0) // ok、const引数
  post(a3 == 0) // ok、参照引数
```

これは、事後条件から使用される非参照非`const`関数引数について、どの時点の値をキャプチャすべきか？という問題を回避するためのものです。

例えば次のようなライブラリ関数があるとします

```cpp
// ユーザーが見る宣言
int generate(int lo, int hi)
  pre (lo <= h)
  post(r: lo <= r && r <= hi);
```

この宣言からは、事後条件によって引数`lo, hi`に対して戻り値が`[lo, hi]`の範囲内に収まっていることが明確に読み取れます。しかしこの関数の実装が次のようになっていたとしたらどうでしょうか

```cpp
// 開発者が見る定義
int generate(int lo, int hi)
  pre (lo <= h)
  post(r: lo <= r && r <= hi) // 事後条件からの引数使用に制限がないとする
{
  int result = lo;

  while (++lo <= hi) // loが更新される
  {
    if (further())
      ++result;      // loよりもゆっくりとインクリメントされる
  }
  return result;
}
```

この関数は値渡しで引数を変更しています。これにより、この定義から見る事後条件の意味は全く違ったものになっています。

関数の非参照引数を変更するというのは正気ならやらないとは思われますが、言語が暗黙的にこれを行う場合があります。

```cpp
std::string g(std::string p)
  post (r: starts_with(p))  // 事後条件からの引数使用に制限がないとする
{
  return p; // 引数の暗黙ムーブ
}
```

関数引数は暗黙ムーブの対象であるため、事後条件でそのような関数引数を使用すると意図通りに動作しなくなります。

このように、関数定義内で引数が変更されていると、関数宣言の契約アサーション（特に事後条件）のみから戻り値に対する推論を行うことができなくなります。このことは事前条件では問題にならず（変更前であるため）、事後条件でのみ問題になります。また、引数が参照である（変更されるものであるため）か`const`である場合（変更できないため）も問題になりません。

これを防止するために、事後条件から使用できる関数引数は参照もしくは非参照の`const`なものに限定されています。

### 宣言と定義が分かれている場合

C++の関数はその宣言と定義を分けて記述することができるほか、宣言だけなら何度も再宣言することができます。この場合にも契約アサーション（事前条件と事後条件アサーション）を指定することはできますが、少し注意が必要になります。

関数に対して事前条件と事後条件アサーションを指定する場合、翻訳単位におけるその関数の最初の宣言に対して指定しておかなければなりません。そして、同じ翻訳単位内で続く再宣言は事前条件と事後条件アサーションの指定について次のどちらかを順守する必要があります

- 事前条件と事後条件アサーションを一切指定しない
- 最初の宣言と全く同一の事前条件と事後条件アサーションを指定する

一つの翻訳単位内でこれが満たされない場合、コンパイルエラーとなります。

```cpp
// 最初の宣言
int f(const int x)
  pre (x != 1)
  post(r : r == x && r != 2);

// 有効な再宣言: 事前/事後条件アサーションを持たない
int f(const int x);

// 有効な再宣言: 同一の事前/事後条件アサーションを持つ
int f(const int x)
  pre (x != 1)
  post(r : r == x && r != 2);

// 有効な再宣言（定義）: 事前/事後条件アサーションを持たない
int f(const int x)
{
  contract_assert(x != 3);  // ok、contract_assertは関係ない

  return x;
}

// 不正な再宣言: 異なる事前/事後条件アサーションを持つ
int f(const int x)
  pre (x != 1);     // ng、post()が足りない

// 不正な再宣言: 異なる事前/事後条件アサーションを持つ
int f(const int x)
  pre (0 < x)
  post(r: r == 0);  // ng
```

後の宣言で事前条件と事後条件アサーションが指定されていない場合、暗黙的に最初の宣言の事前条件と事後条件アサーションと同一のものが指定されているものとして扱われます。

最初の宣言に事前条件と事後条件アサーションが指定されていない場合、あとの宣言で追加することはできません。

```cpp
// 契約アサーションを持たない関数宣言
int f(const int x);

// 不正な再宣言:  事前/事後条件アサーションが追加された
int f(const int x)
  pre (x != 1)
  post(r : r == x && r != 2); // ng
{
  contract_assert(x != 3);

  return x;
}
```

翻訳単位が複数ある場合、ある関数に対応する最初の宣言は翻訳単位毎に1つづつ存在しえます（このような状況はヘッダファイルに契約アサーションを持つ関数の宣言を記述して、そのヘッダを複数のソースファイルでインクルードした時に起こります）。ある関数に対応するそのような複数の最初の宣言の間で同一の事前条件と事後条件アサーションが指定されていない場合、診断不要のill-formed（IFNDR）となります。

```cpp
/// 翻訳単位1におけるf()の宣言
int f(const int x)
  pre (x != 1)
  post(r : r == x && r != 2);

/// 翻訳単位2におけるf()の宣言
int f(const int x)
  pre (x != 1)
  post(r : r == x && r != 2); // ok

/// 翻訳単位3におけるf()の宣言
int f(const int x); // IFNDR、契約アサーションの指定がない
```

IFNDRは診断されない、すなわちコンパイルエラーとなることを要求されていないため、未定義動作となります。これはODR違反が起きているのとほぼ同じことです。ここでの翻訳単位には共有ライブラリの様なものも当然含まれます。

これらのルールを順守するために問題となるのは、事前条件と事後条件アサーションの同一性の定義です。ある2つの事前/事後条件アサーションのシーケンスは次の場合に同一とみなされます

1. 同一の契約アサーション指定（1つの`pre(...)`/`post(...)`）が
2. 同じ順序で指定されている

この一つの契約アサーション指定の同一性は、ODRの意味での同一です。これは、関数宣言`d1`と`d2`がありそれぞれ契約アサーション指定`c1`と`c2`（この`c1. c2`は単一の`pre(...)`/`post(...)`）を持っているとき、その述語`p1`と`p2`は、関数宣言`d1`と`d2`に対応する仮想的な関数本体に配置されコンパイルされたときにODRに違反しない、場合にODRの意味で同一となります。すなわち、この時に仮想的な`d1, d2`の定義が異なっていない状態の場合に、契約アサーション指定`c1`と`c2`は同一となります。

このODRの同一性をより具体的に言えば、この仮想的な`d1, d2`がビット単位で全く同じ機械語コードにコンパイルされてなおかつ同じ動作となる場合、がODRの意味で同一と言えます（最適化の事は考えないでください）。したがって、異なる翻訳単位にある時でも、同じ関数に指定された事前/事後条件アサーションが参照する変数や関数等は同じものにならなければなりません。

この同一性は、C++23時点においてヘッダで定義されている`inline`関数が順守すべきODRのルールとほぼ同じです。

一方で、関数引数の名前やテンプレートパラメータ名、事後条件における結果名、などはここでのODR同一性においては考慮されないため、異なっていても問題ありません。通常の`inline`関数や関数テンプレートでは字句的に同一であることが求められるため、これは通常の関数のODRよりも緩いルールです。

```cpp
/// 翻訳単位1におけるf()の宣言
int f(const int x)
  pre (x != 1)
  post(r : r == x && r != 2);
  
/// 翻訳単位2におけるf()の宣言
int f(const int y)
  pre (y != 1)
  post(b : b == y && b != 2); // ok
  
/// 翻訳単位3
static int x = 0;

/// 翻訳単位3におけるf()の宣言
int f(const int a)
  pre (x != 1)
  post(r : r == x && r != 2); // IFNDR、翻訳単位1と2の契約アサーションと同一ではない
  
/// 翻訳単位4におけるf()の宣言
int f(const int y)
  post(b : b == y && b != 2)
  pre (y != 1);               // IFNDR、順番が異なる
```

この同一性の違反は翻訳単位が同じであればチェックされてコンパイルエラーとなります（異なっている場合はIFNDRです）。

#### 事前条件と事後条件アサーションにおけるラムダ式の使用

ラムダ式がリンケージを必要とする場所（関数宣言付近）に現れる場合、そのクロージャ型は必ず固有のものになるというルールがあります。これと契約アサーション指定の同一性のルールによって、事前条件と事後条件アサーションでラムダ式が使用されている場合は同じ翻訳単位内で全く同じ文字列によって再宣言していたとしても（仮に契約アサーション指定を省略したとしても）、同一性を満たすことができません

```cpp
// 同じ翻訳単位内にあっても
void f() pre([]{ return true; }()); // ok、最初の宣言
void f() pre([]{ return true; }()); // ng、最初の宣言の契約アサーションと同一ではない
void f();                           // ng、最初の宣言の契約アサーションと同一ではない
```

このような場合でも再宣言をしなければ事前条件と事後条件アサーションでラムダ式を使用することに問題はありません。

ラムダ式のこのような性質は、C++20 P0351R4で導入された評価されない文脈でのラムダ式の仕様に基づいています。

#### 事後条件での関数引数の使用

事後条件で関数引数を使用するにはその引数が非参照なら`const`である必要があるというのは説明しました。これは単に事後条件が存在する場合のみ`const`にしておけばいいのではなく、すべての宣言において`const`でなければなりません。通常の関数宣言では、値引数の`const`有無は関数宣言のマッチングにおいて考慮されないため、少し注意が必要です。

```cpp
void f1(int n);
void f1(const int n); // ok、f1()の再宣言

void f1(const int n) post(r: r == n);
void f1(int n);   // ng、事後条件で使用している引数が（非参照）非const
```

再宣言において事前条件と事後条件アサーションが省略された場合、暗黙に最初の宣言と同一の契約アサーション指定がなされているとみなされるため、後の宣言でも事後条件で使用している引数に`const`が必要です。

これはここまでで説明してきたODR同一性とは少し関係がないことですが、契約アサーション付き関数の再宣言においては注意すべきことです。

## 評価セマンティクス

ここまでで、C++ Contractsにおける契約アサーションはどのように記述できるか、ということはなんとなくわかったと思います。すると気になってくることは、この契約アサーションはどう振舞うのか？ということでしょう。

もちろん、契約アサーションは書いて終わりではなく通常のコード同様構文チェックが行われ、契約条件が満たされているかを実行時にチェックすることもできます。契約アサーションがどのように振舞うのか、すなわち契約アサーションのセマンティクスは、大きく評価（条件式のチェック、実行）と評価の後に起こることの2つに分割されて定義されており、どのように評価されるのかについての事をContractsの**評価セマンティクス**と呼びます。

### 4つのセマンティクス

C++ Contractsの評価セマンティクスは単一ではありません。評価するかしないかやどう評価するのか、評価してどうするのかについてはプログラムによって様々な要求・要件があります。言語標準の機能としてそれらの要求をなるべく受け止められるようにするために、評価セマンティクスは固定ではなくいくつか規定されたセマンティクスから選択して適用可能なようになっています。

C++26時点では、4つの評価セマンティクスが標準で規定されています。

1. *ignore*
2. *observe*
3. *enforce*
4. *quick enforce*

これに加えて、実装が実装定義のセマンティクスを使用しても良いことになっています。

これらのセマンティクスは主に、契約条件式を評価（チェック）するかしないか、評価した後に契約違反ハンドラを呼び出すか出さないか、契約違反が起きた後に実行を継続するかどうかが異なっています。

C++26時点の各セマンティクスの違いは表にして一覧できます

|セマンティクス|チェック|違反ハンドラ|違反後継続|
|---|---|---|---|
|*ignore*|しない|呼び出さない|する|
|*observe*|する|呼び出す|する（条件付）|
|*enforce*|する|呼び出す|しない（条件付）|
|*quick enforce*|する|呼び出さない|しない|

表のチェック列の意味は、プログラムの実行が契約アサーションに到達した時にそこに指定された契約条件式を評価するかどうかです。チェックするセマンティクスでは契約条件式が評価されてその結果が`bool`値として取得されます。契約条件式が`false`を返した場合、その契約アサーションの表明する契約が破られている状態であり、これを**契約違反（*contract violation*）**（が発生した状態）と呼びます。

違反ハンドラ列の意味は、契約条件式が評価されて契約違反が発生した後に、契約違反ハンドラと呼ばれる専用のコールバック処理を呼び出すかどうかです。呼び出すセマンティクスでは契約違反発生後に違反ハンドラ（専用のライブラリ関数）を起動して、契約違反をハンドリングします。この違反ハンドラについては次の節で詳しく解説します。

違反後継続列の意味は、契約違反が発生した後でもプログラムの実行を継続するかどうかです。*observe*と*enforce*セマンティクスでは違反後に違反ハンドラが呼ばれてさらにその後に継続するかどうかです。しないセマンティクスでは契約違反が起こるとプログラムが終了されます。

*observe*と*enforce*セマンティクスの継続に関する条件とは、違反ハンドラの呼び出しが正常にリターンするかどうかであり、これらのセマンティクスにおいて違反ハンドラの呼び出し後の振る舞いは次のようになります

- *observe* : 違反ハンドラが正常にリターンした場合、違反を起こした契約アサーションの直後から元の実行パスを再開する
    - 違反ハンドラがリターンしなければ、元の実行パスの再開はされない
- *enforce* : 違反ハンドラが正常にリターンした場合、プログラムを終了する
    - 違反ハンドラがリターンしなければ、プログラムは終了されない

*observe*と*enforce*のデフォルトの継続動作はそれぞれ継続すると継続しないですが、違反ハンドラがリターンするかどうかによってこの動作をオーバーライドできます。これについては次節で詳しく説明します。

*ignore*セマンティクスはチェックを行わないため契約違反が発生しないので違反ハンドラ呼び出しもプログラム終了も行われません。そのため実際に契約違反が発生していたとしてもそれを検出することはできません。しかし、これにより契約アサーションの評価によるオーバーヘッドをゼロにすることができます。なお、契約アサーションが*ignore*セマンティクスでコンパイルし実行される場合でも契約条件式のコンパイル（構文チェック）は行われており、*ignore*セマンティクスのアサーションの契約条件式で参照されているテンプレートは実体化されます（これは`assert`マクロのセマンティクスとは異なる点です）。

*observe*セマンティクスは契約違反をロギングしながらプログラムの実行をなるべく優先するもので、レガシーなコードからの移行期に選択するセマンティクスとなることを目的とするものです。ただし、契約違反が発生した状態でプログラムの実行を継続すると早晩未定義動作に突入するため安全なセマンティクスとは言えない側面があります（これは*ignore*セマンティクスも同様）。

*enforce*セマンティクスは契約違反をロギングしつつも違反時に終了することでバグの影響を最小にとどめることを目的とするものです。契約違反が発生したら速やかにプログラムは終了されるため、未定義動作に陥ることがないという意味で安全なセマンティクスでもあります。このため、*enforce*セマンティクスはデフォルトの評価セマンティクスとしておくことが推奨されています。

*observe*と*enforce*セマンティクスでは契約違反時の診断情報（違反が発生した場所などの情報）が提供されます（これは推奨事項ではあります）。

*quick enforce*セマンティクスは違反ハンドラ呼び出しを伴わず、チェックして違反が発生すると即プログラムを終了させます。これは契約アサーションによって生成されるコードサイズを最小限に抑えるとともに、違反が起きたらなるべく速やかにプログラムを終了させることを目的としています。*quick enforce*では契約違反に関する診断情報が提供されないか最小限のものになる可能性があります。

### プログラムの終了方法

*enforce*/*quick enforce*セマンティクスにおけるプログラム終了方法の選択は実装定義とされつつも、次の3つの方法のいずれかによって行われます

1. `std::terminate()`の呼び出し
2. `std::abort()`の呼び出し
3. プログラム実行の速やかな終了

3が選択される場合は結局その方法は実装定義となるのですが、このような自由度は実行環境とコンパイラにおける様々なユースケースに対応するためのものです。

例えば、`std::terminate()`はプログラムを終了させると言いつつも`std::set_terminate()`で登録されたハンドラを呼び出してから`std::abort`を呼び出すなど少し時間がかかります。`std::abort()`でも`SIGABRT`に対応するシグナルハンドラが登録されているとそちらが呼び出されてしまいます。これらのハンドラで終了が阻まれることはほぼ無いとは思いますが、契約違反が発生している状態のプログラム終了直性で任意のユーザーコードを実行可能な瞬間があります。

3の選択肢は、そもそも`std::abort()`のような機構でさえも利用できない環境であったり、セキュリティを重視して言葉通り即座にプログラムを終了させるオプションを実装が提供することに対応するためのものです。

例えば、Clangは`__builtin_trap()`や`__builtin_verbose_trap()`（デバッガのアタッチに対応している）といった余計なことをほとんど一切行わずに即座にプログラムを終了させる方法を提供しています。*quick enforce*セマンティクスの場合はこうした実装を取ることがその意義からしても合理的です。

また、このプログラム終了の方法はセマンティクス毎に異なることができ、同じセマンティクスであっても状況によって異なることが許可されています（後の方の「契約条件式の評価中の例外」のところで少し触れています）。

### 評価セマンティクスの選択

契約アサーションのデフォルトのセマンティクスとしては*enforce*セマンティクスが標準で推奨されていますが、セマンティクスを選択する方法は実装定義とされています。

これはコンパイラオプションを通してコンパイル時に一括で制御することが想定されていますが、より柔軟にリンク時（異なる設定によってコンパイルされた翻訳単位の結合時）・ロード時（設定ファイルや環境変数など）・実行時（外部のデバッガの指定など）などのタイミングで選択することも許可されています。

全ての契約アサーションのセマンティクスを一括で指定するようなコンパイラオプションはほぼ確実に提供されるため、通常はそれを使用することになると思われます。

評価セマンティクスの選択方法の自由度に対応して、ある契約アサーションの評価にどの評価セマンティクスが選択されるかも実装定義とされます。特に、この評価セマンティクスの選択は契約アサーション毎に異なっていても良く、さらに同じ契約アサーションでも評価ごとに異なるセマンティクスを選択することも許可されています。

例えば、Clangの試験実装では`-fcontract-evaluation-semantic`というオプションによって指定することができ、さらに`-fcontract-group-evaluation-semantic`というオプションで別途属性で指定したグループごとに制御する方法を提供しています。

評価セマンティクスは4つありましたが、実装は4つの評価セマンティクスの全てをサポートしていなくても良いとされています。

### 契約アサーションの評価タイミングと評価回数

チェックされる評価セマンティクスにおいて、契約アサーションの契約条件式の評価のタイミングは厳密に指定されています。

- 事前条件（`pre()`） : 関数呼び出し時、関数引数の初期化後、関数本体の実行前
- 事後条件（`post()`） : 戻り値が初期化され、ローカル変数がすべて破棄された後、関数引数の破棄の前
- アサーション文（`contract_assert()`） : 制御フローがその文に到達した時

ある関数に対して事前条件または事後条件が複数指定されている場合、事前条件と事後条件それぞれの評価の内側で、個別の事前条件/事後条件は宣言順（指定された順序）で評価されます。

```cpp
int f(const int n, int& out)  // (0) : 関数引数初期化
                              // (10) : 関数引数破棄
  pre(n != 0)     // (1)
  post(0 < out)   // (2)
  pre(n < 100)    // (3)
  post(r: n < r)  // (4)
{
  int m = 10; // (5)

  out = g(n, m);

  contract_assert(out != 0);  // (6)
  contract_assert(0 < m);     // (7)

  return m + n; // (8)
} // (9) : ローカル変数破棄
```

この例において、関数`f()`が呼び出されるとまず、事前条件（`(1)(3)`）が次の順序で評価されます

```
(0) => (1) => (3) => (5)
```

次に関数本体が評価され

```
(5) => (6) => (7) => (8)
```

最後に事後条件（`(2)(4)`）が評価されます

```
(8) => (9) => (2) => (4) => (10)
```

ここでの評価とは評価セマンティクスによる選択の前の契約アサーションそのものの評価のことです。各契約アサーションがこの順で評価されていく際に、*ignore*セマンティクスが選択されている契約アサーションではその契約条件式の評価がスキップされます。以下、この節では評価という言葉を専らこの意味で使用しています。契約アサーションの評価とその条件式の評価の違いに注意してください。

コンストラクタ/デストラクタやコルーチンなどにおいては少し異なる点があるものの、この評価順序には自由度はほぼありません。例えば同じ契約アサーションのシーケンス（事前条件や事後条件の集まり）内で順番を入れ替えて宣言順を逸脱することはできません。

ただ、このような評価の順序とは別にそれぞれの契約アサーションが何回評価されるのかは実装定義となります。

ある関数について、次の3種類の**順次評価される（evaluated in sequence）**契約アサーションの集合が形成されます

1. 関数に指定されている事前条件アサーション
2. 関数に指定されている事後条件アサーション
3. 関数本体内で連続しているアサーション文

1と2はある関数に最大1つづつしかありませんが、3は1つの関数に対して複数存在しえます。

先に述べたような評価順序で評価が行われる一方で、このような集合の1つの評価中には、そこに含まれる評価済みの契約アサーションを、実装定義の回数まで任意の評価セマンティクスで再評価することが許可されます。すなわち、契約アサーションは1回以上評価され、契約条件式は0回以上評価されます。そして、この上限は必ずしも1ではありません。

```cpp
void f(int i)
{
  // この2つのアサーション文によって1つの順次評価される集合が形成される
  contract_assert(i > 0);   // #1
  contract_assert(i < 10);  // #2

  // この集合においては次の評価順序と回数は有効（合法）
  // #1 => #2
  // #1 => #1 => #2 => #2
  // #1 => #2 => #1 => #2
  // #1 => #2 => #2 => #1
  
  // 次の評価順序と回数は無効（合法ではない）
  // #2 => #1
  // #1
}
```
```cpp
void f(int *p)
  pre( p != nullptr ) // precondition #1
  pre( *p > 0 );      // precondition #2

// 順次評価される集合は{#1 #2}
// この集合においては次の評価順序と回数は有効（合法）
// #1 => #2
// #1 => #1 => #2
// #1 => #2 => #1 => #2
// #1 => #2 => #2 => #1 

// 次の評価順序と回数は無効（合法ではない）
// #2 => #1
// #2 => #2
// #1
// #1 => #1
```

推奨事項として、コンパイラはこのような契約アサーションの評価の回数を指定するオプションを提供することと、デフォルトの動作は各契約アサーションが正確に1回だけ評価されるようにしておくことが推奨されています。

このような複数回の評価が起こる場合の例として、事前条件/事後条件を持つ関数が共有ライブラリで定義されている場合におけるチェックの実装があります。契約アサーションの指定に関して前述のように最初の宣言に指定されている必要があることから、ヘッダファイル経由でそのような関数に対する事前条件と事後条件は呼び出し側（共有ライブラリ使用者）でも呼び出される側（共有ライブラリ内部）でも同じ契約アサーションが見えています。その場合に呼び出し側と呼び出される側の両方の翻訳単位がチェックされるセマンティクス（例えば*enforce*）を指定してコンパイルされているとします。

この時、実装の取りえる評価方法として、呼び出し側と呼び出される側の双方で契約アサーションの評価およびその条件式の評価を行うという方法があります。この場合、事前条件は関数呼び出し直前に、呼び出し側と呼び出される側の両方で2回評価され、事後条件も同様に2回評価されます。

```cpp
/// ヘッダファイル
int f(int *p)
  pre( p != nullptr ) // #1
  pre( *p > 0 )       // #2
  post(r: 0 < r)      // #3
  post(r: r < 100);   // #4

/// 共有ライブラリcpp
#include "header.hpp"

int f(int *p) {
  // 関数契約に違反しない定義とする
  ...
}

/// 共有ライブラリ利用側cpp
#include "header.hpp"

int main() {
  int n = 100;

  int r = f(&n);
  // 呼び出し側と呼び出される側の両方で契約アサーションを評価する場合
  // #1 #2 |翻訳単位の壁| #1 #2 f() #3 #4 |翻訳単位の壁| #3 #4
  // の順序で契約アサーションが評価される
}
```

このような評価方法は想定され許可されている物であり、なおかつ合理的なものでもあります。リンク時やロード時のセマンティクス選択がサポートされている場合、このような評価を削減できる可能性はありますが、それをサポートするかどうかやどうサポートするかは実装定義となるため、このように評価される可能性を排除するには実装がどうなっているかを調べる必要があります。

それ以外に繰り返し評価が発生するケースとして、アサーションのテストのためにあえて複数回試行する評価方法が挙げられています。

`const`化と合わせてこの契約アサーションの評価回数の不定性によっても、契約条件式が副作用を伴いまたそれに依存するような使い方はC++26 Contractsの使い方としては間違っている物であり、推奨されません。

さらに、コンパイラがある契約条件式についてその評価結果が`true`になるか`false`になるかを証明できる場合、コンパイラはその式を評価する代わりにその結果を使用することが許可されています。別の言い方をすると、その契約条件式（副作用をもちうる）と同じ結果になる副作用の無い式を代わりに使用して契約条件式の評価を行うことができます。あるいは、一度評価した結果を再利用する形で評価を省略することも許可されています。

このような契約条件式の置換は契約条件式の一部分ではなく全体に対して行われるため、契約条件式の持つ副作用はすべてが起こるか全く起こらないかのどちらかになります。

このことからも、契約アサーションの契約条件式には副作用を含めるべきではありません。

### 定数式における契約アサーション

定数式においても契約アサーションは評価されます。そこでも4つの評価セマンティクスのいずれかが適用されることになるのですが、少しだけ実行時と異なる点があります。

まず、コンパイル時には契約違反ハンドラというものがありません。このため、*enforce*と*quick enforce*は同じ意味になります。そして、違反後継続しない（プログラム終了する）セマンティクスにおけるプログラム終了は、コンパイル時においてはコンパイルエラーを意味します。

また、契約違反の定義も変わっており、定数式において契約違反とみなされるのは

- 契約条件式が`false`に評価された
- 契約条件式が定数式で実行できなかった
    - 条件式内にコンパイル時実行できないものがあった

の2つの場合です。特に、後者はコンパイル時特有の違反となります。

これらのことを踏まえて実行時の評価セマンティクスの特性表書き直すと次のようになります

|セマンティクス|チェック|違反時|
|---|---|---|
|*ignore*|しない|何もしない|
|*observe*|する|診断出力|
|*enforce*|する|コンパイルエラー|
|*quick enforce*|する|コンパイルエラー|

*ignore*セマンティクスは実行時と同様に定数式でも何もしないセマンティクスとなります。ただし、指定された契約条件式がコンパイルされないわけではありません（定数式中で評価されないだけです）。

*observe*セマンティクスも実行時と同様に定数式でも違反を検出したらそれについての診断メッセージを発行して実行を継続します。定数式においては未定義動作の心配がないため安全になります。また、この時の診断メッセージは警告として扱われます。

*enforce*と*quick enforce*は前述のようにコンパイル時には同じ意味になり、契約違反を検出するとコンパイルエラーを発生させてコンパイルを停止させます。この時、コンパイルエラーメッセージの一環としてどのアサーションで違反が起きたかなどの診断情報が出力されます。

詳細は後述しますが、実行時の契約違反時の診断メッセージ発行は違反ハンドラの動作の一部として定義されています（このため推奨事項とされます）。一方定数式では違反ハンドラが無いことから、契約違反時の診断メッセージ発行はコンパイラの直接の責務となっており、コンパイル時は診断メッセージの発行が義務付けられています。そのため、チェックされるセマンティクスでは定数式で契約違反が起こると確実に何かしらの診断メッセージが発行されます。

このコンパイル時の評価セマンティクスの選択に関しても実行時同様に実装定義となりますが、特筆すべき点として、同じ契約アサーションが実行時とコンパイル時で異なるセマンティクスによって評価されることが許可されています。対応して、コンパイラは実行時とコンパイル時で異なるセマンティクス指定フラグを提供することができます。

例えば、コンパイル時は*enforce*セマンティクスで評価し（バグをなるべく検出）、実行時は*ignore*セマンティクスで評価し実行する（オーバヘッド最小化）、という選択は合理的なものとなりえます。

#### 契約アサーションとコンセプト/SFINAE

定数式における契約違反の定義に契約条件式が定数式で実行できない場合が含まれているのは、式が定数式で実行できるかどうかによってSFINAEすることを防止するためです。

C++26 Contractsの設計原則の一つとして「コンセプトは契約を認識しない」というものがあり、契約アサーションの存在が（コンセプトだけではなく）コンパイル時のプロパティに影響を与えないように設計されています。

そのため、契約アサーションの存在あるいはその評価セマンティクスの違いによって

- コンセプトが満たされるかどうか
- SFINAEの結果
- オーバーロード解決の結果
- `if constexpr`の分岐結果
- `noexcept`演算子の結果

が変化することはありません。

ある関数に契約アサーションを追加してもその周辺でのコンパイル時のプロパティは追加前と変化せず、評価セマンティクスの選択が変わった時でも変化せず、契約アサーションを取り除いても変化することはありません。

## 契約違反ハンドラ

契約アサーションがどのように評価されるのかは評価セマンティクスによって定義され、制御されることが分かりました。次は、評価された後、特に*observe*と*enforce*の2つのセマンティクスにおいて契約違反が起きた場合にどう振舞うかの部分を見ていきます。

この評価後のセマンティクスは、**契約違反ハンドラ（*contract-violation handler*）**というものによって定義されています。契約違反ハンドラはグローバルに単一かつ共通の契約違反ハンドリング機構であり、契約違反時のセマンティクスをプログラム中で統一的に制御します。

以下本書では契約違反ハンドラの事を単に**違反ハンドラ**と呼びます。また、前述のように違反ハンドラが考慮されるのは実行時のみなので、ここで説明することは基本的に実行時の振る舞いについてです。

### 違反ハンドラの呼び出し

実行時の契約アサーション評価において契約違反が検出された場合、そのセマンティクスによって行われることは次のようになっていました

- *ignore* : 契約違反は検出されない（のでそのまま継続）
- *observe* : 違反ハンドラを呼び出す
- *enforce* : 違反ハンドラを呼び出す
- *quick enforce* : プログラム終了

*observe*と*enforce*セマンティクスでは契約違反時に違反ハンドラを呼び出して処理を委譲します。違反ハンドラはグローバルにインストールされた専用の関数であり、`::handle_contract_violation()`という名前で次のようなシグネチャを持ちます

```cpp
// 違反ハンドラのシグネチャ（グローバル名前空間で定義）
extern "C++" {
  void handle_contract_violation(const std::contracts::contract_violation&) noexcept(/*実装定義*/);
}
```

このシグネチャはあくまで説明のためのものです。また、この関数はグローバルモジュールに属し、C++言語リンケージを持ちます。

この関数の動作の詳細（定義）は実装定義とされますが、引数に提供された`std::contracts::contract_violation`型オブジェクトの持つ契約違反に関する情報を診断情報として出力する実装をデフォルトとすることが推奨されています。前述のように、実行時の契約違反時の診断メッセージ出力はこの違反ハンドラが担っています。

違反ハンドラはかなり特殊な関数であり、`::handle_contract_violation()`という名前を持つことが規定されてはいるもののユーザーがアクセス可能ではないことが規定されています。仮にこの名前でアクセス可能だったとしても、`std::contracts::contract_violation`型のオブジェクトを構築する方法が提供されていないため、結局呼び出すことはできません。

*observe*と*enforce*セマンティクスによる契約アサーションの評価によって契約違反が発生した場合、実装はその契約違反に関する情報を保持する`std::contracts::contract_violation`型オブジェクトを何らかの手段で作成し、それを渡して違反ハンドラを呼び出します。この`std::contracts::contract_violation`型オブジェクトの作成方法は未規定（実装定義）ですが、少なくとも`new`を使用して動的メモリ確保を行わない方法によるものであることは指定されています。

デフォルトの違反ハンドラはこのように受け取った`std::contracts::contract_violation`型オブジェクトから契約違反に関する情報を取得して、何らかフォーマットされた診断情報を標準出力などに出力してから正常に終了（リターン）します。違反ハンドラが正常にリターンした後でどうするかは前述のようにセマンティクスによって異なります

- *observe* : 違反を起こした契約アサーションの直後から元の実行パスを再開する
- *enforce* : プログラムを終了する

このような過程から明らかかもしれませんが、違反ハンドラはシグナルハンドラのようなものではなく、通常の制御フローの延長で呼び出されるものです。あるスレッドで契約アサーションが評価されて違反ハンドラが呼ばれた場合、そのスレッドの実行は違反ハンドラ内に移り、*observe*セマンティクスの場合は違反ハンドラが正常にリターンすると違反が起きた契約アサーションの直後から元のプログラムの実行が再開されます。

違反ハンドラは契約違反時の処理を一元的に担い、管理することを目的としています。そのため、プログラム中のどの場所で契約違反が発生したとしても同じ違反ハンドラ呼び出されます。

### 違反ハンドラの置換

違反ハンドラはユーザーによって上書きして、ユーザー定義の任意の違反ハンドラをインストールする事ができます。ただし、これがサポートされるかは実装定義です。

サポートされる場合、違反ハンドラのインストールはグローバル`new/delete`演算子の置換と同じ方法によって行います。つまり、`::handle_contract_violation()`関数を同じシグネチャでユーザー定義することで行います。以下、違反ハンドラのインストールの事を置換と言います

```cpp
// 違反ハンドラのシグネチャ（グローバル名前空間で定義）
void handle_contract_violation(const std::contracts::contract_violation&) noexcept(/*自由*/);
```

`noexcept`指定に関しては自由度があり、`noexcept(true)`と`noexcept(false)`のどちらを指定しても置換することができます。属性の指定（`[[noreturn]]`など）も自由なほか、違反ハンドラ自体に事前条件と事後条件を指定することもできます。

```cpp
// 有効な違反ハンドラ宣言の例

void handle_contract_violation(const std::contracts::contract_violation&) noexcept {
  ...
}

[[noreturn]]
void handle_contract_violation(const std::contracts::contract_violation&) noexcept(false) {
  ...
}

void handle_contract_violation(const std::contracts::contract_violation&)
  pre(...)
  post(...)
{
  ...
}
```
```cpp
// 無効な宣言の例

// 戻り値は返せない
int handle_contract_violation(const std::contracts::contract_violation&) {
  ...
}

// 引数型はconst参照固定
void handle_contract_violation(std::contracts::contract_violation&) {
  ...
}

// 引数型はconst std::contracts::contract_violation&でなければならない
void handle_contract_violation(const int&) {
  ...
}

// これもダメ（多分
void handle_contract_violation(const auto&) {
  ...
}

// inlineであってはならない
inline void handle_contract_violation(const std::contracts::contract_violation&) {
  ...
}

// C++リンケージを持つ必要がある 
extern "C" void handle_contract_violation(const std::contracts::contract_violation&) {
  ...
}
```

正しいシグネチャで置換した違反ハンドラはリンク時にデフォルトの違反ハンドラからの置換が行われ、実行時にはデフォルトの違反ハンドラと同様にそのプログラム中で一貫して使用されます。

そして、置換するユーザー定義の違反ハンドラ内では通常のC++コードによって違反時処理を記述することができます。どのようにログ出力するのかや何か追加の処理をするのかなどを自由に記述できます。

*observe*と*enforce*セマンティクスではプログラムを終了するか継続するかについて、違反ハンドラが正常にリターンするかどうかによって条件付けされていました。

- *observe* : 違反ハンドラが正常にリターンした場合、違反を起こした契約アサーションの直後から元の実行パスを再開する
    - 違反ハンドラがリターンしなければ、元の実行パスの再開はされない
- *enforce* : 違反ハンドラが正常にリターンした場合、プログラムを終了する
    - 違反ハンドラがリターンしなければ、プログラムは終了されない

ユーザー定義の違反ハンドラでは「違反ハンドラがリターンしなければ」の場合を明示的に選択することができます。

すなわち、ユーザー定義違反ハンドラの動作によって、*observe*/*enforce*セマンティクスにおけるプログラム継続/終了という最後の動作をカスタマイズすることができます。具体的には、例外を投げて契約違反を起こした処理から脱出したり（サポートされる場合のみ）、あるいは違反ハンドラ内で無限ループに入れておくことで契約違反を起こした処理を続行せずにプログラム全体の動作を継続させる、といった実装を取ることができます。

```cpp
#include <contracts>

void handle_contract_violation(const std::contracts::contract_violation& info) noexcept(false) {
  throw make_my_contract_violation_exception(info); // 独自の例外を送出してプログラムのより上位の部分でハンドルする
}

void handle_contract_violation(const std::contracts::contract_violation& info) noexcept {
  for(;;) std::this_thread::yield();  // 無限ループに入れることで元の処理を続行させず、プログラムを終了もさせない
}
```

`contract_violation`クラスも含めて、違反ハンドラ関連のユーティリティ（`std::contracts`名前空間のもの）は`<contracts>`ヘッダで定義されています。契約アサーションを使用するだけならこのヘッダをインクルードする必要はありませんが、違反ハンドラをユーザー定義しようとするとヘッダのインクルードが必要になります。

ユーザー定義違反ハンドラから例外を用いて脱出する場合、*enforce*セマンティクスではプログラムの終了は行われなくなります。例外を用いずとも、違反ハンドラをリターンしなければ*enforce*セマンティクスにおいてはプログラムの終了はされなくなります。

このような実装において、デフォルトの違反ハンドラの行う診断メッセージ発行を再現したい場合があるでしょう。その場合のために、デフォルトの違反ハンドラを呼び出すためのライブラリ関数`std::contracts::invoke_default_contract_violation_handler()`が用意されています。

```cpp
#include <contracts>

void handle_contract_violation(const std::contracts::contract_violation& info) noexcept(false) {
  // ログ出力してから例外送出
  std::contracts::invoke_default_contract_violation_handler(info);
  throw make_my_contract_violation_exception(info);
}
```

`std::contracts::invoke_default_contract_violation_handler()`に違反ハンドラ引数の`contract_violation`オブジェクトを渡して呼び出すことで、違反ハンドラが置換されている場合でもデフォルトの違反ハンドラを呼び出すことができます。

違反ハンドラに提供される`contract_violation`オブジェクトは、違反ハンドラの置換に関わらず同じ方法で作成され供給されます（これをカスタマイズする方法は提供されません）。その生存期間については未規定ですが、少なくとも違反ハンドラが実行を完了するまでの間は生存していることが保証され、またこれは`contract_violation`オブジェクトを介してアクセス可能な全てのオブジェクトについても保証されます。

上記のように例外で脱出する場合、`contract_violation`オブジェクトの参照を保存してもいつまで正しく使用できるかは分からないので注意しましょう。上記の例の場合、`make_my_contract_violation_exception`関数/クラスコンストラクタ内で情報を取り出すだけにとどめておくのが無難です。

このような違反ハンドラの置換は共有ライブラリが動的リンクされている場合でも動作するはずではありますが、どのようにサポートされるかは`new/delete`同様に実装定義となります。特に、共有ライブラリと利用側とで異なるユーザー定義違反ハンドラの置換が行われている場合、おそらくODR違反で未定義動作になります。

### `std::contracts::contract_violation`

違反ハンドラの引数である`std::contracts::contract_violation`クラスは`<contracts>`ヘッダで定義されているライブラリクラスです。違反ハンドラではこのクラスを介して契約違反に関連した情報を取得することができます。

まず初めにですが、このクラスはユーザーによって構築することができません。デフォルトコンストラクタもコピー/ムーブコンストラクタも代入演算子もすべて利用できません。基本的に実装が作成して違反ハンドラに渡してきた参照を介して利用するのみです。この性質のため、このクラスオブジェクトを使用する場所は実行時の違反ハンドラ内部に限られます。

`std::contracts::contract_violation`クラスの宣言はおおむね次のようになっています

```cpp
namespace std::contracts {
  class contract_violation {
    // no user-accessible constructor
  public:
    // コピー/ムーブ禁止
    contract_violation(const contract_violation&) = delete;
    contract_violation& operator=(const contract_violation&) = delete;

    // デストラクタはvirtualであってもよい（実装定義）
    /*実装定義*/ ~contract_violation();

    // 契約違反を起こした述語のテキスト表現を取得する
    const char* comment() const noexcept;

    // 契約違反の種類を取得する
    contracts::detection_mode detection_mode() const noexcept;
    
    // 現在の呼び出しの違反ハンドラの終了後に終了するかどうかを取得する
    bool is_terminating() const noexcept;
    
    // 違反ハンドラを呼び出した契約アサーションの種類を取得する
    assertion_kind kind() const noexcept;
    
    // 契約違反を起こした契約アサーションのソースコード情報を取得する
    source_location location() const noexcept;
    
    // 契約違反を起こした契約アサーションの評価セマンティクスを取得する
    evaluation_semantic semantic() const noexcept;
  };

  // デフォルトの違反ハンドラを呼び出す
  void invoke_default_contract_violation_handler(const contract_violation&);
}
```

`detection_mode`、`assertion_kind`、`evaluation_semantic`は独自の列挙型であり、その定義は次のようになっています

```cpp
namespace std::contracts {

  // 契約アサーションの種類を表す
  enum class assertion_kind : /*未規定*/ {
    pre = 1,
    post = 2,
    assert = 3  // contract_assertに対応
  };

  // 評価セマンティクスの種類を表す
  enum class evaluation_semantic : /*未規定*/ {
    ignore = 1,
    observe = 2,
    enforce = 3,
    quick_enforce = 4
  };

  // 契約違反の種類を表す
  enum class detection_mode : /*未規定*/ {
    predicate_false = 1,
    evaluation_exception = 2
  };
}
```

コメントにも簡単に記載していますが、`std::contracts::contract_violation`クラスの各メンバ関数は次のようなものです

- `.comment()` : 契約違反を起こした契約条件式を文字列化したものが取得できる
    - 通常の文字列リテラル（`"str"`）と同じエンコーディングでnull終端された文字列が得られる
- `.detection_mode()` : 違反ハンドラを呼び出した契約違反の種類を表す列挙値が取得できる
- `.is_terminating()` : 現在の違反ハンドラを正常にリターンしたらプログラムが終了されるかどうかが取得できる
    - 標準の4つの評価セマンティクスの中では、*enforce*セマンティクスの場合のみ`true`を返す
- `.kind()` : 契約違反を起こした契約アサーションの種類を表す列挙値が取得できる
    - `pre()`/`post()`/`contract_assert()`の3種類に対応している
- `.location()` : 契約違反を起こした契約アサーションのソースコード情報が取得できる
    - 可能な場合、契約違反を起こしたのが事前条件であるならば、返されるソースコード情報は関数呼び出しを行った場所に関するものになる
- `.semantic()` : 契約違反を起こした契約アサーションの評価セマンティクスの種類を表す列挙値が取得できる

自明かもしれませんが、これらのメンバ関数によって得られるのは現在の違反ハンドラを呼び出した契約違反に関するものです。ある違反ハンドラの呼び出し時の`contract_violation`オブジェクトが、その呼び出しとは無関係な別の契約違反に関する情報を持つことはありません。

```cpp
#include <contracts>
#include <print>

// ユーザー定義違反ハンドラにおけるcontract_violationの利用例
void handle_contract_violation(const std::contracts::contract_violation& info) noexcept(false) {
  using namespace std::contracts;

  const auto loc = info.location();

  std::println("{} In function '{}'", loc.file_name(), loc.function_name());
  std::print("{}:{}:{}: contruct violation occured", loc.file_name(), loc.line(), loc.column());
  std::println(" [assertion={}, semantic={}]", info.kind(), info.semantic());
  std::println("{}", info.comment());

  if (info.is_terminating()) {
    throw make_my_contract_violation_exception(info);
  }
}
```

`.comment()`/`.location()`関数の返す値は正確には実装定義とされており、上記で説明したような値を返すことは推奨事項となっています。これは、セキュリティが重要なプログラムやリバースエンジニアリング防止を意図するプログラムにおいて、リリースバイナリにオリジナルソースに関する情報を残さないようにすることをサポートするためです。その場合、空文字やデフォルト構築された`source_location`オブジェクトが返されます。

なお、`contract_violation`クラスも含めた`<contracts>`ヘッダ全体はフリースタンディング環境でも利用可能です。Contracts機能そのものが実行環境を問わずに利用可能であることが強く意図されています。

### 契約条件式の評価中の例外

契約違反の種類を表す列挙型`contracts::detection_mode`に`evaluation_exception`という値があるのに気づいたかもしれません。ここまでは契約違反の定義を契約条件式が`false`を返した場合としていましたが、正確には実行時の契約違反の定義は次のようになっています

- 契約条件式が`false`を返した
- 契約条件式の評価中に例外が送出された

すなわち、違反ハンドラは契約条件式が`false`を返す以外にも、契約条件式が例外を送出した場合にもその例外をハンドリングする形で呼び出されます。

```cpp
bool condition() noexcept(false);

void f() pre(condition());

int main() {
  // condition()が例外を送出すると契約違反が発生する
  // observe/enforセマンティクスの場合、違反ハンドラが呼び出される
  // quick enforceセマンティクスの場合、プログラムが終了される
  f();
}
```

*quick enforce*セマンティクスの場合でも契約条件式からの例外送出は契約違反として扱われ、即座にプログラムが終了されます。*quick enforce*セマンティクスでは例えば、暗黙の`noexcept`コンテキスト中で契約条件式を評価することで例外送出時に直ちに`std::terminate`につなげることで`try-catch`を省略するという実装が許可されています。

*observe*/*enforce*セマンティクスで評価された場合に違反ハンドラでハンドルされた例外は違反ハンドラによって処理されたものとして扱われるため、違反ハンドラが正常にリターンした後に再スローされることはありません。例外オブジェクトは違反ハンドラ終了とともに破棄され、*observe*セマンティクスの場合は通常通りに実行を継続します。

ユーザー定義違反ハンドラ内では、契約違反を起こした例外オブジェクトを`std::current_exception()`などによって取得することができます。

```cpp
#include <contracts>
#include <exception>

void handle_contract_violation(const std::contracts::contract_violation& info) noexcept(false) {
  using namespace std::contracts;

  if (info.detection_mode() == detection_mode::evaluation_exception) {
    // 独自にハンドルする
    my::handle(std::current_exception());

    // あるいは再スローする
    std::rethrow_exception();
  }
}
```

`contract_violation`クラスの`.detection_mode()`による契約違反原因の識別はこのために提供されています。

### 違反ハンドラからの例外送出と`noexcept`

違反ハンドラの置換時に違反ハンドラからの例外送出を認めるかどうかは実装定義とされています。サポートされる場合、ユーザー定義の違反ハンドラはほぼ自由に例外を送出することができます。この違反ハンドラの送出する例外の扱いは次のようになります

- 違反ハンドラが送出した例外は契約アサーションが指定されている関数本体から送出されたものとして扱われる
- 関数が`noexcept`指定されている場合、通常通り`std::terminate`で終了する

事前条件と事後条件でも、そこからの例外はその関数内から投げられたものとして扱われるため、その関数が`noexcept`指定されていると`std::terminate`で終了します。

これにより、違反ハンドラの置換とユーザー定義違反ハンドラからの例外送出がサポートされている場合、契約アサーションが存在する場合は常にそこから例外が送出される可能性が生じることになります。これについては標準化の過程でも議論の的ではありました。しかし、違反ハンドラから例外を送出する場合は稀であることサポートが必須ではないことによって許容されています。このようなユースケースとして、プログラムの継続が何より優先される環境において、契約違反をハンドリングしたうえで例外によってより上位で復帰を行う場合が主なものとして提示されていました。

違反ハンドラから例外を送出することを選択する場合は、関数に`noexcept`を付加するかどうかについてLakos Rule（狭い契約と`noexcept`についてのルール）に従っておくようにする必要があります。すなわち、何らかの契約（事前条件）を持つ関数は無条件`noexcept`すべきではありません。これをプログラム全体で徹底しなければ違反ハンドラからの例外送出そのものによってプログラムが終了されてしまいます。

違反ハンドラから例外を送出することを許容する場合、契約アサーションが指定されている関数の`noexpcet`性のクエリ結果（`noexcept`演算子の結果）がどうなるべきかという点も問題になります。この問題は解決が難しかったため、C++26では契約アサーションに対する`noexpcet`性クエリを禁止することによって対処しています。

`noexcept`演算子は式が例外を送出するかを取得するものであるため、`noexcept(pre(...))`のようなクエリはできません（`pre()`が式ではないため）。そして、関数呼び出し式が例外を送出するか（`noexcept(f())`が`true`となるか）はその関数（`f`）が`noexcept`指定されているかによって決まり、より大きな式の場合でも基本的にその式で呼ばれる関数の`noexcept`指定の有無によって決まります。このため、`f`に契約アサーションが指定されていてもその存在はクエリ結果に影響を与えません。

そのうえで、次の制限を加えることで契約アサーションの例外送出について`noexcept`演算子でクエリすることができないようにされています

- `contract_assert`が文であることで、`noexcept(contract_assert(false))`のようなクエリができない
- 特殊メンバ関数の`default`宣言における契約アサーションの指定を禁止することで、`default`実装における`noexpcet`性の決定に契約アサーションが関与しない

関数の特殊メンバ関数の（暗黙の）`default`宣言は、言語が自動で`noexcept`性のクエリを行う唯一の場所です。このことは特殊メンバ関数の`default`宣言に契約アサーションを指定できない理由の一つでもあります。

### 契約アサーション評価の疑似コード

ここまで説明してきた契約アサーションの評価から違反ハンドラのリターンによるプログラム終了/続行の過程に関しては、通常のC++を用いた疑似コードで記述することができるので、参考にここに載せておきます。

```cpp
// 評価セマンティクスの取得
evaluation_semantic _semantic = __current_semantic();

if (evaluation_semantic::ignore == _semantic) {
  // ignoreセマンティクスの場合
  // なにもしない
} else if (evaluation_semantic::observe == _semantic
        || evaluation_semantic::enforce == _semantic
        || evaluation_semantic::quick_enforce == _semantic)
{
  // 契約チェックを行うセマンティクスの場合

  if consteval {
    // 定数式における契約チェック
    ...
  } else {
    // 契約条件式の結果を保存する変数
    bool _violation;
    // 違反ハンドラを呼び出すべきかを表す変数
    bool _handled = false; // Violation handler has been invoked.

    // 契約条件式のチェックと違反ハンドラの呼び出し(必要な場合）
    try {
      // 契約条件式の評価
      _violation = __check_predicate(X);
    } catch (...) {
      // 契約条件式の評価が例外を送出した場合
      if (evaluation_semantic::quick_enforce == _semantic) {
        // quick enforceセマンティクスの終了処理
        std::terminate(); // implementation−defined program termination
      } else {
        // 例外送出による契約違反ハンドリング
        _violation = true;

        // 契約違反ハンドラの呼び出し
        __handle_contract_violation(_semantic, detection_mode::evaluation_exception);

        _handled = true;
      }
    }

    if (_violation && evaluation_semantic::quick_enforce == _semantic) {
      // quick enforceセマンティクスの終了処理
      __builtin_trap(); // implementation−defined program termination
    }
    if (_violation && !_handled) {
      // 契約違反ハンドラの呼び出し
      __handle_contract_violation(_semantic, detection_mode::predicate_false);
    }
    if (_violation && evaluation_semantic::enforce == _semantic) {
      // enforceセマンティクスの終了処理
      std::abort(); // implementation−defined program termination
    }
  }
} else {
  // その他実装定義のセマンティクスの処理
}
```

実際に契約アサーションごとにこのようなコードが挿入されるわけではなく、細部をどう実装するかについても実装の自由度がありますが、起こることを理解するのには役立つと思われます。

## その他の関数に対する契約アサーション

ここまででC++26 Contractsの基本的な部分の解説は完了していますが、ここまでの例はほぼすべて普通の関数で例示していたため、普通ではない関数と契約アサーションがどのように相互作用するのかについてを引き続いて見ていきます。

なお、ここの節での契約アサーションとは、ほぼ事前条件と事後条件の事です。`contract_assert`は文が書ける場所には書くことができて、その外側の関数がなんであるかに関わらず使用することができるためです。

### 仮想関数

C++26 Contractsでは仮想関数に対する契約アサーションの指定はサポートされておらず、`virtual`な関数の契約アサーションを指定するとコンパイルエラーになります。

C++26 Contractsの提案段階ではかなり直前のリビジョンまで残っていたのですが、継承時の契約アサーションの扱いとその評価方法について合意を得ることができず、C++29以降の機能として検討していくためにC++26では禁止されました。

### ラムダ式

ラムダ式に対しても契約アサーションを指定することができます。指定する場所も通常の関数と同様に本体の直前、`requires`の後です。

```cpp
[](const int x) -> int
  pre (x != 1)
  post(r : r == x && r != 2)
{
  contract_assert(x != 3);

  return x;
};
```

このように指定された事前条件と事後条件は、クロージャ型の関数呼び出し演算子（`operator()`）に指定されたものとして扱われます。そのため、その他のルールも通常の関数とほぼ同様になります。

ラムダ式特有の注意点は、契約アサーションからラムダ式外部の変数を参照しているときでも、それが暗黙キャプチャをトリガーしないことです。したがって、契約アサーションだけで使用されており、ラムダ式内の他の部分で使用されていない外部変数の使用はコンパイルエラーとなります。

```cpp
static int i = 0;

void test() {
  auto f1 = [=] pre(i > 0) {  // ok、グローバル変数の参照（キャプチャ不要）
  };

  int i = 1;

  auto f2 = [=] pre(i > 0) { // ng、iを暗黙キャプチャしない
  };

  auto f3 = [i] pre(i > 0) { // ok、iは明示的にキャプチャされている
  };

  auto f4 = [=] {
    contract_assert(i > 0); // ng、iを暗黙キャプチャしない
  };

  auto f5 = [=] {
    contract_assert(i > 0); // ok、iは契約アサーション以外で暗黙キャプチャされている
    (void)i;
  };

  auto f6 = [=]
    pre([]{
        bool x = true;
        return [=]{ return x; }(); // ok、xは暗黙キャプチャされている
      }()
    ){
  };
}
```

ラムダ式の契約アサーションから外部の変数を参照しキャプチャが必要な場合は、明示的にキャプチャするか、ラムダ式の他のところで参照するようにしましょう。

これは、契約アサーションの存在がクロージャ型のコンパイル時プロパティに影響を与えないようにするためにこうなっています。

### コンストラクタ/デストラクタ

コンストラクタ/デストラクタに対しても契約アサーション（事前条件と事後条件）を指定できます。ただし、通常の関数とは異なり、コンストラクタ/デストラクタはクラスのメンバの初期化と破棄を担うものであるため、評価のタイミング等について注意すべき部分があります。

特に、コンストラクタの事前条件およびデストラクタの事後条件の評価タイミングは次のようになります

1. コンストラクタの事前条件 : コンストラクタ引数の初期化後、すべての非静的メンバが初期化される前
    - 関数`try`ブロックやメンバ初期化子リストを含めた関数本体の評価前
2. デストラクタの事後条件 : すべてのサブオブジェクト（基底クラスおよび非静的メンバ）が破棄された後

したがって、これらの場所の契約アサーションからそのクラスのメンバにアクセスすると未定義動作となります。特に、派生クラスの一部としてコンストラクタ/デストラクタが実行されている場合、派生クラスオブジェクトは初期化前/破棄後でありいずれにせよ生存期間外にあるため、`dynamic_cast`や`typeid`等動的型に依存する処理も未定義義動作となります。

これらの場所の契約アサーションにおけるメンバアクセスリスクを軽減するために、これらの場所では非静的メンバ名を直接使用できないようになっています。どうしてもアクセスしたい場合は`this->`を明示することでアクセス可能ですが、この場合でも安全に使用可能なのはそのアドレスくらいです。

```cpp
struct X {
  int i = 0;

  X()
    pre (i == 0)          // ng、メンバ変数名を使用できない
    pre (this->i == 0)    // UB、メンバ変数は初期化前
    pre (check(&this->i)) // ok、メンバ変数のアドレスは使用可能
    post (i == 0)         // ok、コンストラクタ事後条件は通常通り
  {}

  void f()
    pre (i == 0)    // OK
    post (i == 0);  // OK

  ~X()
    pre (i == 0)              // ok、デストラクタ事前条件は通常通り
    post (i == 0)             // ng、メンバ変数名を使用できない
    post (this->i == 0)       // UB、メンバ変数は破棄済
    post (check(&this->i));   // ok、メンバ変数のアドレスは使用可能
};
```

これ以外の部分は通常の関数の契約アサーションと同じです。

### `default`/`delete`な関数

ほとんどの場合コンストラクタやデストラクタが該当しますが、最初の宣言で`default`指定されている関数に対する事前条件/事後条件の指定は禁止されています。

```cpp
struct X {
  X() pre (true) = default; // ng、defaultな関数にpre()を指定できない

  X& operator=(X&) pre(true) = default; // ng
};

struct Y {
  Y() pre (true); // ok
};


Y::Y() pre (true) = default;  // ng、最初の宣言と異なる契約アサーションが指定されている
```

代入演算子や比較演算子など他の`default`定義が可能なものでも同様です。

これは特殊メンバ関数のトリビアル性と契約アサーションの評価で矛盾が生じることを回避するためであるほか、`default`な関数はコンパイラによって生成されるデフォルトのインターフェースを持つものとみなされますが、契約はそのデフォルトのインターフェースの一部になりえない（デフォルト実装が無い）ためです。

これと同様に、`delete`指定されている関数宣言に対する事前条件/事後条件の指定も禁止されています。

```cpp
struct X {
  X() pre (true) = delete; // ng
};
```

こちらはコンストラクタやデストラクタだけでなく通常の関数でもです。

### コルーチン

コルーチンにも契約アサーション（事前条件と事後条件）を指定することができます。ただし、それはコルーチンそのもの（サスペンド時やその解除時、コルーチンの終了時など）に対するものではなく、ランプ関数と呼ばれるコルーチンを起動する関数に対する契約になります。

コルーチンのランプ関数とは、コルーチンになっている関数を呼び出した際にまず呼び出される、コルーチンを起動するための関数です。このランプ関数はコルーチンステートを初期化してコルーチン起動して最初のサスペンド地点まで進めてから、その関数の戻り値型として表示されているコルーチン型（コルーチンハンドルを保持している型）のオブジェクトを返して終了します。

コルーチンに対する事前条件と事後条件はこのランプ関数に対する事前条件と事後条件となります。したがって、コルーチンに対する事前条件と事後条件はユーザーが記述したコルーチン処理の`co_yield`や`co_return`に適用されるわけではありません。

```cpp
// sequenceはコルーチンだとする
std::generator<int> sequence(int from, int to)
  pre (from <= to);

// このようなコルーチンとその契約は次のように書き換えた関数に対する契約アサーションとほぼ等しい

// コルーチン実装詳細
std::generator<int> sequence_impl(int from, int to);

// 非コルーチン、疑似ランプ関数
std::generator<int> sequence_ramp(int from, int to)
  pre (from <= to)
{
  return sequence_impl(std::move(from), std::move(to));
}
```

この例のコルーチン`sequence()`には事前条件が指定されていますが、それは通常の関数によって書きかえた後の`sequence_ramp()`の様な関数に対して同じ契約アサーションを指定している場合と同じように扱われます（あくまでこれはイメージです）。

コルーチンに対する事前条件と事後条件の評価の順序は次のようになります

- 事前条件 : 関数引数の初期化後、コルーチンステートの初期化処理の前
    - 特に、関数引数がコルーチンステートに格納される前
- 事後条件 : コルーチン戻り値の`return`時
    - ランプ関数の`return`時
    - コルーチン状態内のあらゆるものの破棄との間での順序は不定

コルーチンの事後条件では、コルーチン関数の戻り値（`co_yield`した値ではなく）に対して事後条件を指定することができます

```cpp
awaitable<int> cancelable_session(int id)
  pre (0 < id)
  post (r: is_cancelable(r)); // rはawaitable<int>オブジェクト
```

標準のコルーチン型（`std::generator`、`std::execution::task`）はいずれもこのような状態をチェックする関数が無いのでできることはほぼ無いですが、ユーザー定義のコルーチン型では有用な場合もあるでしょう。

コルーチンではコルーチンステートの初期化時にその関数引数をすべて、値の場合は`const`修飾を無視して、コルーチンステート内にムーブして格納します。このため、事後条件から関数引数を参照することが安全ではなく、禁止されます。

```cpp
std::execution::task<int> coro(const std::string str)
  pre(str.empty() == false)   // ok、事前条件からの関数引数の使用は安全
  post(str.empty() == false); // ng、コルーチン事後条件において関数引数を使用できない
```

`contract_assert()`にはこれらのような制限がなく（引数の使用もコルーチンステート内の物を参照するので無問題）、コルーチン内部で使用できコルーチンの制御フローの一部としてアサーションが評価されます。ただし、契約条件式で使用できる式について少し制限があり、契約条件式でawait式/yield式を使用できません。

```cpp
std::generator<int> f() {
  contract_assert(((co_yield 1), true)); // ng
}

stdex::task<void> g() {
  contract_assert(0 < (co_await query_database())); // ng

  ...
}
```

ただしこれは、`contract_assert()`が置かれている直接のコルーチンに対して影響を与えるawait式/yield式が禁止されているだけで、そうでない場合は使用可能です。例えば、コルーチンになっているラムダ式の即時呼び出しを契約条件式内で呼ぶ場合などです

```cpp
contract_assert(([]()-> std::generator<int> {
  co_yield 1; // ok
}(), true));
```

これらの式の存在が関数をコルーチンにするため、これは`contract_assert()`に対する制限というよりはawait式/yield式に対する制限ではあります。この制限は、契約アサーションの評価セマンティクスの選択がコルーチンの動作を変更しないようにするためのものです。

### 関数ポインタ

C++26 Contractsでは関数ポインタ/メンバ関数ポインタに対する契約アサーション（事前条件と事後条件）の指定はサポートされず、指定するとコンパイルエラーになります。

```cpp
typedef int (*fpt)(int) post (r: r != 0); // ng

int f(int x)
  post (r: r != 0);

int (*fp)(int) post (r: r != 0) = f;  // ng
int (X::*fptr)(int) post (r: r != 0) = &X::f; // ng
```

契約アサーションの指定はその型に影響を与えないため、どのようにアサーションをポインタに適用し、どのようにそれを伝播するのかなどの設計の検討がほとんど進んでいないためです。これも、C++29以降の機能として予定されています（が、おそらく現在の関数ポインタにそのまま指定できるようにはならないでしょう）。

ただし、契約アサーションを持つ関数を関数ポインタに入れることはでき、関数ポインタ経由の呼び出し時にも契約アサーションは評価されます。

```cpp
int f(int x)
  pre(0 < x)
  post (r: r != 0);

int (*fp)(int) = f; // OK

fp(-1); // 契約違反が発生する
```

これはメンバ関数ポインタや`std::function`等ラッパを介した場合でも同様で、契約アサーションを持つ関数がそれらに格納されている場合、呼び出し時に契約アサーションは評価されます。

### `main()`

`main()`関数にも事前条件と事後条件を指定することができます。

```cpp
int main(int argc, char *argv[])
  pre(2 < argc)
  post(r: r != 0)
{
  ...

  return 10;
}
```

評価等のルールにも特別なところはなく、通常の関数と同じように扱われます。

この例のようにプログラム引数に対して使用する以外にも、`main()`呼び出し前（あるいは終了前）のプログラムのより大域的な状態を検査するのにも使用できます。

### 未定義動作とObservable Checkpoints

C++26ではObservable Checkpointsという概念が導入されています。これはUBによるタイムトラベル最適化（コード上の因果関係を逆転するような最適化）を防止するためのチェックポイントであり、プログラムのある地点で発生したUBがObservable Checkpointsを超えてプログラムをUBにすることを防止するものです。

`std::observable()`およびI/O関数の呼び出しがObservable Checkpointsとして扱われますが、契約アサーションもObservable Checkpointsを導入します。

より正確には契約アサーションに関連して次のイベントがObservable Checkpointsとして扱われます

- 契約アサーション評価に際して、契約条件式の評価が開始されたこと
- *observe*セマンティクスで評価された契約アサーションから呼び出された違反ハンドラが正常にリターンしたこと

これによって、契約アサーション以降で発生したUBが契約アサーションを超えて波及し、契約アサーションそのものを消し去ったりその評価をスキップすることが禁止されます。

```cpp
volatile int i = 0;

void f(int *p) {
  if (p != nullptr) // #1
  {
    ++i;
  }

  contract_assert(0 <= *p); // p == nullptrの場合UB
}
```

この例では、`#1`の後の`contract_assert`が*ignore*セマンティクス以外で評価される場合、（Observable Checkpointsがないとすると）タイムトラベル最適化により`#1`のチェックが削除されるなどの事が発生しえます。

`contract_assert`の契約条件式の評価開始地点がObservable Checkpointsになっていることで、契約アサーションの条件式にUBが含まれていたとしても、その契約アサーションよりも前にそのUBが波及するのを阻止します。

```cpp
void g(int *p) {
  contract_assert(p != nullptr); // #2

  ++(*p); // p == nullptrの場合UB
}
```

この例では、`#2`の`contract_assert`が*observe*セマンティクスで評価されている場合、違反ハンドラの呼び出しがリターンした後でも実行が継続されることでその後の`*p`が実行されUBになります。、（Observable Checkpointsがないとすると）タイムトラベル最適化により`#2`の契約アサーションそのものが削除される可能性があります。

*observe*セマンティクスによって呼び出された違反ハンドラのリターンがObservable Checkpointsになっていることで、契約アサーション以降のUBが契約アサーションそのものを削除することを阻止します。このことは*observe*セマンティクスが契約違反発生後も実行継続する際のUBの影響を緩和します。

### 標準ライブラリの堅牢化モード（Standard Library Hardening）

C++26では標準ライブラリに対して堅牢化モードというものが定義され、標準ライブラリの一部の事前条件が実行時にチェックされ、違反していたらプログラムが終了されるようになります。

この堅牢化モードにはC++26 Contractsが使用され、Contractsの枠組みの中で標準ライブラリの実行時検査が行われます。

```cpp
int main() {
  std::vector vec = {1, 2, 3};

  int& r = vec[4];  // 堅牢化モードでは契約違反が発生し、プログラムが終了される
}
```

堅牢化モードにおける評価セマンティクスは*enforce*/*quick enforce*のどちらかが使用され、非堅牢化モードでは*ignore*セマンティクスが使用されます（*observe*セマンティクスは基本的に使用されません）。

### リフレクション

C++26 Contractsの設計原則として契約アサーションの存在が他のもののコンパイル時プロパティに影響を与えないというものがあることもあり、関数に指定された契約アサーション（事前条件と事後条件）をリフレクション（`^^`）を使用して取得する方法はありません。これはリフレクションを用いて関数をコピーする際にコピーできないプロパティがあるということでもあります。

一応検討はされているようですが、どうなるかはまだ分かりません。

ただし、契約アサーション内でスプライス式を使用した時でも`const`化が適用されるようにはなっています。

```cpp
void f(int p)
  pre(( ++p, true))       // ng、pはconst化されている
  pre(( ++[:^^p:], true)) // ng、`[:^^p:]`もconst化されている
{}
```

より正確には、スプライス式が契約アサーション内で変数または構造化束縛のリフレクション値に対して適用されているとき、そのリフレクション対象の変数が契約アサーション外部で宣言されている場合にその型に対して`const`を追加します。

# P2900 Contractsの設計に関して

C++26 Contracts機能として採択された提案文書はその文書番号からP2900と呼称されることがあります。P2900は議論の過程で確立されたいくつかの設計原則を骨子として前章で説明したような機能を設計しており、その設計原則を知ることでC++26 Contracts機能がなぜこうなっているのか、どういう意図があるのか、どう利用すべきなのかを理解することができます。

P2900にはその議論の過程で確立された16の設計原則があり、大きく5つのカテゴリに分かれています。

1. 契約アサーションは冗長な検査であるということ
2. 副作用に関する原則
3. 契約アサーションと自然言語による契約
4. 拡張性についての原則
5. 互換性についての原則

これらの原則について見ていく前に、P2900の設計方針の出発点と基本方針についてまず見てみます。

## 出発地点

C++にContracts機能を導入するという議論が始まった際に最初に提示された目標は、プログラマが関数契約に関する1つのアノテーションをコードに追加するだけで、次の2種類の正当性検証ツールを有効化することでした。

1. Cの`assert`マクロのように、実行時に（オプショナルな）検査を行う機能
2. （理想的にはローカルな範囲で実行可能な）正当性検証のための静的解析機能

これらの2つは本質的に異なるツールではあるものの同じアノテーション形式を利用することができます。これらを実現するためには、両機能の典型的な利用方法を本来あるべき形から変更したり何らかの妥協が必要でした。典型的には次のような点があります

1. アサーションを関数宣言に追加する必要がある
2. アサーションには`bool`に変換可能なC++の式を指定する必要がある

契約アサーションがマクロや関数の様な既存の言語構造ではなく専用のアノテーションとして新しい言語機能になっているのは、特にアサーションを静的解析につなげるためです。すなわち、任意のツールが共通で利用可能な、1つに標準化され広く認知されている注釈のための表記法を用意するためです。

C++ Contracts機能は`assert`マクロのような既存の慣行を標準化するだけのものではなく、オプショナルな実行時検査と静的解析の間に相乗効果を生み出そうとしています。これには、単なる実行時検査ツールでは必要のない新たな課題が伴います。

しかしこれによって、C++プログラマは契約アサーションを1つ追加するだけで、それに基づいた複数の方法でプログラムの正当性検証を行うことができる、というメリットが得られます。

## MVP（Minimum Viable Product）

P2900はC++20での失敗（プロセス終盤での機能取り下げ）を受けて、Contracts機能について合意可能かつ実用的な最小範囲の機能セットをまずC++に導入するMVP（Minimum Viable Product）と呼ばれる方針を取っています。

MVPは、C++20 Contractsに対して異論の大きかった機能や議論と合意形成に時間のかかる機能を避けて、Contractsの基本機能と合意形成が容易な部分を最初のC++ Contracts機能として導入することを目指すものです。議論に時間がかかる機能については、追加の機能として後（C++29以降）で導入していくことにしています。

MVPはまた、設計原則の確立とともにC++ Contractsの基礎固めとその後の方向性を定めることで、Contracts機能の継続拡張・改善のための出発点となることも意図しています。

従って、C++26 ContractsはC++ Contractsの完成形ではありません。契約アサーションの評価セマンティクスの明示指定や仮想関数に対する契約指定のサポートなどはC++29の機能として導入することを予定しています。

## 契約アサーションは冗長な検査であるということ

P2900設計原則の1つめのカテゴリです。

Contracts機能の目的は、プログラムが正しいかどうかをコード上で識別可能にすることにあります。そのため、Contracts機能を利用することそのものによってプログラムの正当性が変化してしまうと機能そのものの存在価値が無くなります。したがって、Contracts機能の設計においてはそうならないことを大原則としています。

### 設計原則1: 基本原則

> プログラム内に契約アサーションが存在することによって、プログラムの正当性を変化させてはならない。

### 設計原則2: 冗長性の原則

> 正しいプログラムにおいて、契約アサーションは冗長なものである。  
> すなわち、契約アサーションの一部または全部を削除したとしても、そのプログラムの正当性は変化しない。

### 設計原則3: コンセプトは契約を認識しない

> 関数やコードブロックに契約アサーションが存在するという事実だけで、コンセプトの充足可能性、オーバーロード解決やSFINAEの結果、`if constexpr`で選択される分岐、あるいは`noexcept`演算子の返す値、が変化してはならない。

この原則は特に、定数式における契約アサーションの仕様に表れています。定数式における契約違反はSFINAE可能なエラーにはならず、契約アサーションの存在が他のコンパイル時のC++プロパティに影響を与えないようにされています。

### 設計原則4: ゼロオーバーヘッド原則

> 実際にチェックが行われない（つまりignoreセマンティクスの）契約アサーションの存在は、プログラムの動作に影響を与えてはならない。  
> 例えば、追加のラムダキャプチャを誘発してクロージャオブジェクトのメンバを追加するような事態は避けるべき。

この原則は、ignoreセマンティクスの選択が`assert`マクロのようにチェックを無かったことにしないという仕様に表れています。ignoreセマンティクスの選択された契約アサーションは実行時に評価されなくなるだけで、ODRの観点からは無視されません。

### 設計原則5: 選択されたセマンティクスの独立性

> 特定の契約アサーションの評価においてどの評価セマンティクスが選択されるか、およびそのセマンティクスがチェックを行うものであるかどうかは、一般にコンパイル時に検出可能であってはならない。  
> このような検出が行われると、契約チェックが有効化された場合に動作が変化するプログラムが実行される可能性がある。

原則3~5は原則1と2をより具体化したもので、ここで本質的に重要な原則は1と2です。

契約アサーションは契約プログラミングを実践しようとして新しく問題になることをチェックするものではなく、既存のプログラムにすでにあった契約や不変条件を表明しチェックするためのものです。したがって、契約アサーションは（正しいプログラムでは）プログラマがすでに対処済みの事について改めて表明するものであり、そのチェック対象は仕様そのものになります。このような意味で、契約アサーションによる検査は本質的に冗長な検査となります。

そのため、そのような冗長な検査を追加したとしてもそれをきっかけとしてプログラムの動作が変化し、正しさが壊れることがあってはなりません。

これらの原則はContracts機能の設計の基盤となっている原則であり、様々な部分がこの原則を基礎としています。

C++26契約プログラミング機能はこれら5つの原則をかなり重視して設計されているものの、原則に違反するようなプログラムを完全に禁止できているわけではありません。しかし、これらの原則に違反するようなプログラムを記述することは推奨されず、プログラマはそれを行うべきではありません。これらの原則は設計がそうなっているということだけでなく、利用者であるプログラマもそのように利用すべきということも言っています。

## 副作用に関する原則

前節の原則は主に契約アサーションが存在することそのものの影響についてのものでした。そのうえで、契約アサーションが実行時に評価される際にも同様に、そのこと自体がプログラムの正当性を損ねてはなりません。

その存在や評価によってプログラムの正しさが変化してしまうような契約条件式（契約述語）を「破壊的述語（destructive predicate）」と呼び、プログラムの正しさを変化させてしまうような副作用を「破壊的副作用（destructive side effects）」と呼びます。プログラムの正しさを効果的に検証するためには、破壊的述語を排除する必要があります。

### 設計原則6: 破壊的副作用の禁止

> 評価時にプログラムの正当性に影響を及ぼしうる述語を持つ契約アサーションは、サポートされるべきではない。

### 設計原則7: 契約アサーションの完全性

> 個々の契約アサーションは、自然言語で記述された契約条項に対する完全な検証をカプセル化するものである。

契約アサーションの完全性とは、ある1つの契約（仕様）のチェックを行う場合は1つの契約アサーションだけでチェックが完結していることです。例えば、1つの契約事項をチェックするために複数の契約アサーションを用いる（必然的に何らかの状態に対する副作用を持つ）ことは認められません。

この原則により、契約アサーションはそれが何を検査しているのかについて単独かつローカルに推論することができます（他のアサーションと比較しなくてもそのチェック対象は明確になる）。

### 設計原則8: 契約アサーションの独立性

> ある契約アサーションの評価結果は、他のいかなる契約アサーションの評価結果にも影響を及ぼしてはならない。

### 設計原則9: 契約アサーションの評価の独立性

> ある契約アサーションの評価結果は、同一の契約アサーションに対する後続の（2回目以降の）評価結果に影響を及ぼしてはならない。

原則8と9によって、プログラム全体のグローバルなContracts設定がプログラムにどのような影響を与えるかを予測可能にします（すなわち、動作を変更したりしないこと）。契約アサーションが評価されるされないによってプログラムの正しい動作が変更されるべきではありません。

ここでのこれらの原則が言うことは、契約アサーションに指定される契約条件式（契約述語）は契約アサーションも含めたプログラムの動作を変更するような副作用（破壊的副作用）を持つべきではない、ということです。これは契約アサーションによるプログラムの正しさの検証に無くてはならない前提です。

しかし、コンパイラは破壊的副作用を検出できず、したがって破壊的述語をコンパイルエラーにすることはできません。また、単に副作用を持つというだけでは（それが破壊的かどうか分からないため）未定義動作となるわけでもありません。

これらの原則を達成するために、契約条件式における`const`化や契約条件式からの例外が違反ハンドラ呼び出しに変換されるという機能や、契約述語の評価回数の不定性といった性質が導かれています。`const`化は副作用を、例外の変換は契約アサーションが追加されたことによって関数から例外が送出されるような事態を防止し、契約述語の評価回数が厳密に規定されていないことによって副作用の発生を正当性の担保にできないようにしています。

とはいえ、そのような防御策は完全ではなく抜け道があります。そのためここでの原則も、利用者が意識してこれらの原則に反するような利用を行わないようにすべきものであり、それはContracts機能を用いてプログラムの正しさを向上させようとするならば必須のことです。

前節のコンパイル時の契約アサーションに関する原則と同様に、ここでの実行時評価に関する原則もContracts機能の設計において極めて重要な基盤となっています。

## 契約アサーションと自然言語による契約

契約アサーションがチェックするのは従来自然言語で指定されていた契約（仕様）の一部であり、契約アサーションを使用するために新しくチェックしなければならないことが増えるわけではありません。

### 設計原則10: 契約アサーションは自然言語による契約を検証する

> 関数契約アサーションの評価は、それが付随する関数の評価と結びついていなければならない。  
> そうすることで、そのアサーションは他の関数ではなくその関数の自然言語による契約（またはその一部）を検証することになる。

### 設計原則11: 関数契約アサーションは呼び出し側と呼び出される側の両方で有用である

> 関数契約アサーションは呼び出し側（caller）と呼び出される側（callee）の双方にとって極めて重要な意味を持つ。  
> 特に、呼び出し側と呼び出される側の間の契約として、次の2つの約束が交わされる  
> 1. 呼び出し側は、関数の事前条件を満たすことを約束する。これにより、呼び出される側はこれらの事前条件が常に満たされるものとして処理を行うことができる  
> 2. 呼び出される側（関数の実装者）は、適切に呼び出された際に関数の事後条件を満たすことを約束する。これにより、呼び出し側はこれらの事後条件が常に満たされるものとして処理を行うことができる

これは契約による設計の基本事項です。この原則はまた、関数契約アサーションは関数のインターフェースの一部であり実装の一部でもある、という主張でもあります。

### 設計原則12: 契約アサーションは制御フローとは異なる

> 契約アサーションはアルゴリズムとしての正当性を検証する手段を提供するものの、その構文構造に関連する特定の実行時動作を保証するものではない。

原則1,2でも述べられていたように、契約アサーションは本質的に冗長なチェックです。そしてそれは、関数のエラー処理の様な通常の制御フローの一部であるような動作とは明確に異なります。もし関数に特定の状態をエラーとして報告するという契約上の要件があるのであれば、その処理はC++の標準的な制御文で行うべきであり、契約アサーションを使ってはなりません。

これらの原則に従う形で、契約アサーションの評価セマンティクスは切り替えが可能になっており、契約アサーションは関連する条件を検証して違反があればプログラムを終了させることもあれば、別のビルドでは何もしないこともあります。

契約アサーションのこのような性質は、翻って契約アサーションをエラー処理や入力検証に使用してはならない理由にもなっています。

## 拡張性についての原則

P2900はC++ Contracts機能の最初の提案ではありますが、最後の提案となろうとしたものではありません。その設計は当初からMVP（Minimum Viable Product）として合意可能な最小の部分をまずC++に導入することを目的としていました。

したがって、P2900の設計作業は合意形成が間に合わないことによって将来的な機能拡張とすることを決定している部分が多々あります。そういった部分や既存機能との相互作用における難しい問題点などの設計上の未解決の課題においては、規格の保証として未定義動作を回避してContracts機能を使用するプログラムの動作が適格（well-defined）であることを優先しています。

### 設計原則13: すべての新しい動作を明示的に定義する

> Contracts機能は、契約アサーションの評価時に新しい未定義動作を明示的に導入することは無い。

とはいえ、契約アサーションの評価中に現行のC++において未定義動作となる式に遭遇した場合は未定義動作が発生します。契約アサーションの中から実行されたからと言って既存のプログラムの動作が変化するわけではありません。

### 設計原則14: 柔軟な進化を可能にするために、不適格を選択する

> Contracts機能が対処すべき問題に対する適切な解決策について明確な合意が形成されていない場合、関連する構成要素は不適格として扱われる。

この選択により、設計上の未解決の部分に未規定（unspecified）や未定義動作（undefined behavior）を割り当てるのではなく不適格（Ill-Formed）としておくことで、規格に準拠した拡張機能による様々な選択肢の検討が可能になり、将来的にC++標準に取り込まれうる解決策の可能性を排除しないようにしています。

ただし、実装定義（implementation defined）の選択は未規定や未定義動作とは異なるもので、Contracts機能の多様な環境におけるサポートのために明確な意図をもって選択されています（詳細は後述）。

## 互換性についての原則

既存のプログラムに契約アサーションを追加する際に、現場での導入を著しく妨げるような不具合（破壊的変更）が引き起こされてしまうとContracts機能の普及が妨げられることになります。そのため、そのような機能にならないようにする必要があります。

### 設計原則15: クライアント側の言語の破壊的変更の回避

> 既存の関数`f()`について、`f()`に契約指定子を追加後も`f()`の定義がコンパイル可能であるならば、既存の正しい`f()`の使用法は引き続きコンパイル可能であり、正しく動作しなければならない。

関数の正しい使用法とは、関数契約に違反するような形での呼び出しを伴わない全ての使用を指しています。それは関数を呼び出すことだけでなく、アドレスを取ることや関数ポインタに代入することやオーバーライドすることなども含まれています。そうした通常の関数の使用に著しい不具合や不整合を引き起こさないようになっています。

### 設計原則16: ABI破壊の回避

> 仕様に準拠した実装は、既存の関数に契約指定子を追加しても、ABIの後方互換性が維持されることを保証する必要がある。

ABIの保護はContracts機能だけではなくC++11以降のほぼすべてのC++機能において強く意識し、また要請されていることです。

例えば、翻訳単位を挟んで呼び出される関数の契約チェックが重複しうる（呼び出し側と呼びだし先の両方でチェックされる）という仕様は、チェックの責務を厳格にしないことによって翻訳単位ごとに独立してその決定を行えるようにすることで、ABI互換性を維持しながら契約チェックを導入するためのものです。

## 実装定義

C++26 Contractsの仕様には実装定義（*Implementation defined*）あるいは推奨（*Recommended practice*）という言葉が多く使用されています。動作を突き詰めていくとその最後の部分は実装定義とされていることが多く、これはC++26 Contractsの提案P2900に対して度々投げかけられていた批判でもありました。

ただ、この実装定義は標準が何も規定していないとか保証していないということではありません。本書の1章とこの章で解説しているようなC++ Contractsの理念や目的は提案文書等を通じて説明されており、標準がその動作やセマンティクスを実装任せにしているというわけではありません。。

なぜ実装定義などの曖昧な規定が多く使用されているのかというと、それはContracts機能をC++が使用可能な場所の全てとは言わないまでもなるべく多くの環境で使用できるようにするためです。

C++の実行環境は、指先サイズのマイコンからスーパーコンピューターまで多様なハードウェアに及んでおり、そこで動作するプログラムに対する要件も様々なものがあります。ある環境では実行リソースが潤沢である程度リッチな機能を実行できる一方で、別の環境では厳しくリソースが制限され標準出力すら持たないことがあります。またある環境ではプログラムの実行停止を可能な限り回避することが求められることもあれば、セキュリティの目的のために実行時エラーが検出されたら最速でプログラムを終了する必要があることもあります。

C++26 Contractsの動作、とくに契約アサーションのセマンティクスの細部を細かく規定すると、それによってContractsを使用できなくなる実装が発生します。上記例に挙げたように相反する要件がある場合もあります。そういった実装の事を顧みずに仕様を厳密に固めてしまうと、それによって実装できなくなる環境が確実に出てきてしまい、そのような場所ではContracts機能を永遠に使うことができなくなります。

それは、Contracts機能の普及を妨げることに繋がり、Contracts機能の目的であるC++プログラムの正しさの向上が成し遂げられなくなります。

Contracts機能は強く待ち望まれている機能でもあり、それはC++プログラムの実行環境に依存した要求ではありません。標準仕様としてContractsを特定の実行環境でしか実装できなくなるようにしてしまえば、それはC++に対する失望を招きかねません。

C++26 Contracts仕様にある実装定義とは、こうした多様な実装・環境・要件をなるべく受け止めるための抽象化層です。無計画に実装定義としているわけではなく、Contractsが全体としてどういう機能であるかということはしかっりと説明・規定した上で、最後の最も実装に近い部分においての振る舞いについてを実装定義としておくことで様々な要件等を吸収しようとしています。

したがって、C++26 Contracts仕様の実装定義とされている部分は、仕様の空白地帯や将来の検討のための設計空間などではなく任意のC++プログラム×プラットフォームに対して単一の回答が有効な解決策とはなりえないような場所です。これは、コンパイラやツールチェーンが特定のユーザーが必要とする機能を提供するための機会となるものです。

また、P3321R0という提案においてP2900 Contractsで実装定義とされている部分について実装が選択可能なオプションが列挙されています。
